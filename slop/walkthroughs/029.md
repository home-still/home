## Improvement #1: Edge Weight Implementation

### Status: OPTIONAL (Performance/Accuracy Enhancement)

### The Theory

**Paper's Equation 10** defines weight multipliers that vary based on:
1. **Semantic label** (CrossLayout, Title, Vision, Regular)
2. **Element orientation** (Horizontal vs Vertical)

The paper mentions **wedge** (edge weights) that adjust based on whether an element is horizontally or vertically oriented.

### Current Implementation

**File:** `xycut-plus-plus/src/utils.rs`, lines ~46-54

```rust
let label = masked.semantic_label();
let (mult_w1, mult_w2, mult_w3, mult_w4) = match label {
    SemanticLabel::CrossLayout => (1.0, 1.0, 0.1, 1.0),
    SemanticLabel::HorizontalTitle => (1.0, 0.1, 0.1, 1.0),
    SemanticLabel::VerticalTitle => (0.2, 0.1, 1.0, 1.0),
    SemanticLabel::Vision => (1.0, 1.0, 1.0, 0.1),
    SemanticLabel::Regular => (1.0, 1.0, 1.0, 0.1),
};
```

**Limitation:** Uses semantic label only, doesn't detect actual orientation from dimensions.

### The Enhancement

Detect element orientation from its aspect ratio and apply appropriate weights.

### Mental Model

**Horizontal elements** (width > height):
- Text lines, section titles
- Flow left-to-right
- Emphasize horizontal ordering (phi4)

**Vertical elements** (height > width):
- Sidebar text, vertical labels
- Flow top-to-bottom
- Emphasize vertical continuity (phi3)

### Implementation

**File:** `xycut-plus-plus/src/utils.rs`
**Location:** In `compute_distance_with_early_exit`, after calculating dimensions

**Step 1: Detect Orientation**

After line ~38 (where `max_dim` is calculated), add:

```rust
let mw = mx2 - mx1;
let mh = my2 - my1;
let max_dim = mw.max(mh);

// Detect element orientation from aspect ratio
let is_horizontal = mw > mh;
let aspect_ratio = if mh > 0.0 { mw / mh } else { 1.0 };

// Consider "clearly horizontal" if width is 2x+ height
// Consider "clearly vertical" if height is 2x+ width
let strongly_horizontal = aspect_ratio > 2.0;
let strongly_vertical = aspect_ratio < 0.5;
```

**Step 2: Apply Orientation-Aware Weights**

Replace the current `match label` (lines ~46-54) with:

```rust
// Equation 10: Semantic AND orientation-specific multipliers
let (mult_w1, mult_w2, mult_w3, mult_w4) = match (label, is_horizontal) {
    // CrossLayout: Orientation doesn't matter (always wide spanning)
    (SemanticLabel::CrossLayout, _) => (1.0, 1.0, 0.1, 1.0),

    // HorizontalTitle: Adjust if actually vertical
    (SemanticLabel::HorizontalTitle, true) => (1.0, 0.1, 0.1, 1.0),  // As expected
    (SemanticLabel::HorizontalTitle, false) => (1.0, 0.5, 0.5, 0.5), // Unexpected vertical

    // VerticalTitle: Adjust if actually horizontal
    (SemanticLabel::VerticalTitle, false) => (0.2, 0.1, 1.0, 1.0),   // As expected
    (SemanticLabel::VerticalTitle, true) => (0.5, 0.5, 0.5, 0.5),    // Unexpected horizontal

    // Vision: Detect table vs figure orientation
    (SemanticLabel::Vision, true) if strongly_horizontal => (1.0, 1.0, 0.5, 0.1),  // Wide table
    (SemanticLabel::Vision, false) if strongly_vertical => (1.0, 0.5, 1.0, 0.1),   // Tall figure
    (SemanticLabel::Vision, _) => (1.0, 1.0, 1.0, 0.1),              // Square-ish

    // Regular: Standard weights
    (SemanticLabel::Regular, _) => (1.0, 1.0, 1.0, 0.1),
};
```

**Step 3: Apply to Both Functions**

The same change should be applied to both:
1. `compute_distance_with_early_exit` (lines ~25-118)
2. `compute_distance` if it still exists

### Expected Behavior Change

**Before:**
- A title labeled "HorizontalTitle" always uses (1.0, 0.1, 0.1, 1.0)
- Even if the actual element is vertical (rare but possible)

**After:**
- A horizontal title uses (1.0, 0.1, 0.1, 1.0) ✓
- A vertical title mislabeled as "HorizontalTitle" uses (1.0, 0.5, 0.5, 0.5) (balanced weights)

### Testing

Create test cases with:
1. Normal horizontal titles → Should behave same as before
2. Vertical sidebar text → Should use vertical-friendly weights
3. Wide tables → Should emphasize horizontal proximity
4. Tall figures → Should emphasize vertical continuity

### Impact

**Pros:**
- More robust to mislabeled elements
- Better handling of unusual layouts (vertical text, rotated elements)
- Closer to paper's theoretical model

**Cons:**
- Slight complexity increase
- May need tuning on real-world documents

**Recommendation:** Implement if you encounter documents with vertical text or need maximum paper accuracy.

## Improvement #2: Phi3 Refinement

### Status: VERIFICATION NEEDED

### Current Implementation

**File:** `xycut-plus-plus/src/utils.rs`, lines ~100-111

```rust
// Component 3 (ϕ3): Vertical continuity
let phi3 = if is_cross_layout {
    -my2 // Cross-layout: prefer above (negative y means higher up)
} else {
    // For titles/figures: prefer regular elements BELOW them
    if ry1 >= my2 {
        ry1 - my2 // Regular below masked - distance is good
    } else {
        1000.0 // Regular above masked - heavy penalty
    }
};
```

### Paper's Description

**From Equation 8:**
> ϕ3: Vertical continuity component

The paper doesn't give an exact formula, but describes the intent:
- For **cross-layout** elements: Prefer placement near the top
- For **single-column** elements: Baseline alignment with text flow

### Potential Issues with Current Implementation

**Question 1:** Should we use `ry1 - my2` or `ry1 - my1`?

```rust
// Current: Uses bottom of masked element
ry1 - my2  // Distance from masked bottom to regular top

// Alternative: Uses top of masked element
ry1 - my1  // Distance from masked top to regular top (baseline alignment)
```

**Baseline alignment** typically aligns tops of elements, not bottoms.

**Question 2:** Is the penalty value (1000.0) appropriate?

- Too low: Elements above might still be chosen
- Too high: Could overflow when combined with other components

**Question 3:** Should cross-layout use `-my2` or consider the regular element position?

```rust
// Current: Only considers masked element position
-my2  // Prefer masked elements higher on page

// Alternative: Consider relationship to regular element
if my1 > ry2 {
    -ry2  // Masked is below regular
} else {
    ry1   // Regular is below or overlaps
}
```

### Recommended Refinement

**File:** `xycut-plus-plus/src/utils.rs`, lines ~100-111

**Replace with:**

```rust
// Component 3 (ϕ3): Vertical continuity
let phi3 = if is_cross_layout {
    // Cross-layout: Prefer elements above current position
    // If masked element is below regular, penalize more
    if my1 > ry2 {
        // Masked is below regular - less desirable for cross-layout
        my1 - ry2
    } else {
        // Masked is above or overlaps - prefer higher position
        -my2
    }
} else {
    // Single column: Prefer elements below (reading flow)
    // Use top-to-top distance (baseline alignment)
    if ry1 >= my2 {
        // Regular is below masked - good for reading flow
        ry1 - my1  // Changed from ry1 - my2 for baseline alignment
    } else {
        // Regular is above masked - heavy penalty
        (my2 - ry1) * 10.0  // Scaled penalty instead of fixed 1000.0
    }
};
```

**Key Changes:**
1. Cross-layout: Consider relationship, not just position
2. Single-column: Use top-to-top (baseline) alignment
3. Penalty: Scaled based on distance, not fixed value

### Verification Steps

1. **Test with cross-layout spanning titles:**
   - Should prefer placement near top of page
   - Should handle both above and below regular text

2. **Test with single-column titles:**
   - Should match with text immediately below
   - Should strongly avoid text above

3. **Compare distances:**
   - Print phi3 values for different scenarios
   - Verify they make semantic sense

### Testing Code

Add temporary debug output:

```rust

    let val = if my1 > ry2 { my1 - ry2 } else { -my2 };
    eprintln!("[PHI3 DEBUG] CrossLayout: my1={}, my2={}, ry2={}, phi3={}",
              my1, my2, ry2, val);
    val
} else {
    let val = if ry1 >= my2 {
        ry1 - my1
    } else {
        (my2 - ry1) * 10.0
    };
    eprintln!("[PHI3 DEBUG] SingleCol: my1={}, my2={}, ry1={}, phi3={}",
              my1, my2, ry1, val);
    val
};
```

Run on test documents and verify values are reasonable.

### Impact

**Pros:**
- More accurate baseline alignment
- Better cross-layout handling
- Scaled penalties instead of arbitrary constants

**Cons:**
- Needs validation against paper (formula not explicitly given)
- May require tuning on real documents

**Recommendation:** Start with current implementation (it works well). Only refine if you encounter specific layout issues or want maximum theoretical accuracy.

### Search and Replace Strategy

**Safe approach:** Use your editor's search and replace:

1. Search for: `pub struct XYCut\b` (regex)
   - Replace with: `pub struct XYCutPlusPlus`

2. Search for: `impl XYCut\b` (regex)
   - Replace with: `impl XYCutPlusPlus`

3. Search for: `XYCut::new` (literal)
   - Replace with: `XYCutPlusPlus::new`

4. Search for: `use.*XYCut[^P]` (regex)
   - Replace with: corresponding XYCutPlusPlus import

**Dangerous:** Don't use blanket search/replace for just "XYCut" as it might catch:
- XYCutConfig (should NOT change)
- XYCutPlusPlus (would become XYCutPlusPlusPlus!)
- Comments mentioning "XY-Cut" algorithm
