# Letterbox Preprocessing Refactor

**Goal:** Extract common image preprocessing logic (resize + pad with aspect ratio preservation) into a shared module to eliminate duplication between `layout.rs` and `ocr.rs`.

---

## Problem Statement

Both the layout detector and OCR detector do the same thing:
1. Resize image to fit inside a target size (1024×1024 or 960×960)
2. Maintain aspect ratio
3. Pad with white to fill target dimensions
4. Track transformation metadata (offsets, scales) for coordinate mapping

**Current state:** This logic is duplicated in two places with slightly different implementations.

**Issue discovered:** The layout detector was using `resize_exact()` which DISTORTS aspect ratio, causing bounding boxes to be incorrectly positioned.

---

## Solution: Letterbox Preprocessing Module

**Why "letterbox"?**
- Standard term in YOLO/computer vision for aspect-ratio-preserving resize with padding
- Same technique as movie letterboxing (black bars to preserve aspect ratio)
- Immediately recognizable to CV engineers

## Key Lessons

**1. Coordinate Space Transformations**
When transforming between coordinate spaces:
- Original image → Resized image → Padded canvas → Model output
- Must reverse each transformation in order
- Offsets are translations, scales are multiplications
- Order matters: translate THEN scale (or in reverse: scale THEN translate)

**2. DRY Principle**
When you see the same pattern repeated:
- Identify what's truly common (the algorithm)
- Identify what varies (the parameters)
- Extract to a shared function/module
- Pass the variations as parameters

**3. Naming Matters**
- Use industry-standard terms when they exist ("letterbox")
- Name things by their purpose, not implementation
- Rust modules are lowercase_with_underscores
- Rust types are PascalCase
