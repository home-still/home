# Walkthrough 036: Fixing Catastrophic PaddleOCR Quality

## Mental Model

OCR quality depends on three factors in order of importance:
1. **Right model for your language** - Chinese models don't learn English word boundaries
2. **Sufficient input resolution** - Small fonts need 300+ DPI, not 150 DPI
3. **Proper preprocessing** - Width limits, normalization, aspect ratio preservation

Your current output:
```
Expected: "Freud S. The origins of psychoanalysis"
Got:      "FreuDSTheonginsotpsvchoanavsis"
```

Shows TWO distinct failures:
- **Missing spaces** (language model mismatch)
- **Character corruption** (resolution/preprocessing issues)

Post-correction can't fix these. You need better OCR output first.

## Build Order

1. **Increase DPI** (30 seconds, immediate impact)
2. **Switch to English recognition model** (10 minutes, fixes spacing)
3. **Add width clamping** (2 minutes, prevents distortion)
4. **Add word segmentation fallback** (5 minutes, safety net)

## Problem Analysis

### Issue 1: Chinese Model on English Text

**Evidence:**
```bash
# Your current models
ch_PP-OCRv4_det_infer  # Detection - OK (language-agnostic)
ch_PP-OCRv4_rec_infer  # Recognition - WRONG for English
```

The `ch_` prefix = Chinese-optimized. Chinese text has no spaces between characters. The model learned:
- Continuous character streams
- Chinese character set with limited English
- No English word boundary patterns

**Why it fails on English:**
```
"The origins" → Model sees continuous chars → "Theonginsot"
```

The model doesn't know English words should be separated.

### Issue 2: Insufficient Resolution

**Your current setting** (`src/pipeline/processor.rs:37`):
```rust
let pages = self.pdf_parser.parse_to_pages(pdf_path, 150)?;
```

**Math:**
- Reference section text: ~8pt font
- At 150 DPI: 8pt × 150/72 = 16.7 pixels tall
- After letterbox resize to height 48: severe upscaling artifacts
- Result: Blurry characters that look like "psvchoanavsis"

**Standard DPI guide:**
- 150 DPI: Large text (14pt+), titles
- 200 DPI: Normal text (10-12pt)
- 300 DPI: Small text (8-10pt), references
- 600 DPI: Very small or degraded text

### Issue 3: Excessive Width Without Clamping

**Your debug output:**
```
Box 0: (16.1, 10.7) -> (468.2, 20.1)  // 452×10 pixels
```

**Current preprocessing** (`src/models/ocr.rs:preprocess_for_recognition`):
```rust
let target_height = 48;
let aspect_ratio = image.width() as f32 / image.height() as f32;  // 452/10 = 45.2
let target_width = (target_height as f32 * aspect_ratio) as u32;  // 48 × 45.2 = 2169
```

Result: **2169×48 image** fed to recognition model.

**PaddleOCR recognition models typically expect:**
- Height: 48 (fixed)
- Width: 10-640 (variable, but limited)

At 2169 pixels wide:
- Extreme aspect ratio distortion
- Potential model instability
- Memory/compute waste

### Issue 4: No Fallback for Spacing

Even with an English model, OCR can fail. You need algorithmic word segmentation as a safety net.

## Step 3: Add Width Clamping

**Impact:** Prevents aspect ratio distortion, improves stability

**Change:**

```rust
// src/models/ocr.rs
// In preprocess_for_recognition method

fn preprocess_for_recognition(&self, image: &DynamicImage) -> Result<ndarray::Array4<f32>> {
    let target_height = 48;
    let aspect_ratio = image.width() as f32 / image.height() as f32;

    // BEFORE:
    // let target_width = (target_height as f32 * aspect_ratio) as u32;

    // AFTER: Clamp to PaddleOCR's expected range
    let target_width = ((target_height as f32 * aspect_ratio) as u32)
        .clamp(10, 480);  // Min 10px, max 480px

    // Rest of preprocessing...
    let resized = image.resize_exact(target_width, target_height, FilterType::Lanczos3);
    // ...
}
```

**Why 480?**
- PaddleOCR recognition models are trained with variable width
- Typical max: 320-640 depending on model
- 480 is safe middle ground

**Note:** Clamping changes aspect ratio. That's OK - the model handles moderate distortion better than extreme width.

**Test:** Check debug output. Widths should now be ≤480.

## Step 4: Add Word Segmentation Fallback

**Impact:** Algorithmically inserts spaces when OCR doesn't

Even with English models, OCR can fail. Add `wordninja` for safety:

### Add Dependency

```toml

```

### Create Segmentation Module

```rust
// src/text_correction/segmentation.rs (NEW FILE)

use wordninja::WordNinja;

pub struct WordSegmenter {
    segmenter: WordNinja,
}

impl WordSegmenter {
    pub fn new() -> Self {
        Self {
            segmenter: WordNinja::new_from_builtin(),
        }
    }

    /// Segment concatenated text into words
    /// Example: "thequickbrownfox" → "the quick brown fox"
    pub fn segment(&self, text: &str) -> String {
        self.segmenter.split(text).join(" ")
    }

    /// Check if text needs segmentation (heuristic: long words without spaces)
    pub fn needs_segmentation(&self, text: &str) -> bool {
        // If average word length > 15 chars, probably concatenated
        let words: Vec<&str> = text.split_whitespace().collect();
        if words.is_empty() {
            return false;
        }

        let avg_length: f32 = words.iter().map(|w| w.len()).sum::<usize>() as f32 / words.len() as f32;
        avg_length > 15.0
    }
}
```

## Known Dragons

### Dragon 1: Model Download Failures

PaddleOCR's CDN can be slow or fail.

**Fix:** Use mirror:
```bash
# If main CDN fails, try GitHub mirror
wget https://github.com/PaddlePaddle/PaddleOCR/releases/download/v2.7.0/en_PP-OCRv3_rec_infer.tar
```

### Dragon 2: paddle2onnx Version Compatibility

Different paddle2onnx versions produce incompatible ONNX.

**Fix:** Use opset 11 (most compatible):
```bash
paddle2onnx ... --opset_version 11
```

### Dragon 3: Dictionary Mismatch

Using Chinese dictionary with English model = garbled output.

**Fix:** Always match dictionary to model:
- Chinese model → ppocr_keys_v1.txt
- English model → en_dict.txt

### Dragon 4: Memory Explosion at 300 DPI

Doubling DPI = 4× pixel count = 4× memory.

**Fix:** Process fewer pages at once, or add batch size control:
```rust
// Process in batches of 5 pages
for chunk in pages.chunks(5) {
    // process chunk
}
```

## Alternative: Switch to Tesseract

If PaddleOCR still fails after all fixes, consider Tesseract:

**Pros:**
- Better English support out-of-box
- More mature for Latin scripts
- Handles word boundaries correctly

**Cons:**
- Slower than PaddleOCR
- Less accurate on modern clean text
- No CUDA acceleration (CPU only)

**Quick test:**
```bash
tesseract input.png output --psm 6 -l eng
```

If Tesseract gives better results, consider switching engines.
