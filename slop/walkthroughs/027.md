# XY-Cut++ Bug Fix Walkthrough: The phi3 Vertical Continuity Bug

## The Problem

**Symptom:** Titles appeared grouped together at the start of the reading order instead of appearing in their natural positions on the page.

**Example:**
```
Expected order:
1. Title (y=215)
2. Text (y=301)
3. Title (y=315)
4. Text (y=334)
...
12. Title (y=950)
...

Actual (broken) order:
1. Title (y=215)
2. Title (y=950)   ❌ Should be #12
3. Title (y=1441)  ❌ Should be #23
4. Text (y=301)
...
```

## Root Cause Analysis

### 1. The Debug Output Revealed the Issue

```
[MERGE] Inserting masked ID 15 (y=215.0) at position 0 (before regular ID 16)   ✓ Correct
[MERGE] Inserting masked ID 24 (y=315.6) at position 2 (before regular ID 25)   ✓ Correct
[MERGE] Inserting masked ID 18 (y=950.6) at position 1 (before regular ID 16)   ❌ WRONG!
[MERGE] Inserting masked ID 17 (y=1441.4) at position 2 (before regular ID 16)  ❌ WRONG!
```

**Pattern:** All 4 titles matched with ID 16, the first regular element. But titles at y=950 and y=1441 should match with elements much lower on the page.

### 2. The Distance Metric

The 4-component distance metric (Equation 8) calculates:
```
D = w₁·ϕ₁ + w₂·ϕ₂ + w₃·ϕ₃ + w₄·ϕ₄

where:
  ϕ₁ = Intersection constraint (overlap penalty)
  ϕ₂ = Boundary proximity (edge distance)
  ϕ₃ = Vertical continuity (y-position relationship)  ← THE BUG IS HERE
  ϕ₄ = Horizontal ordering (x-position)
```

### 3. The Buggy Code

**File:** `xycut-plus-plus/src/utils.rs`
**Function:** `compute_distance_with_early_exit`
**Lines:** ~100-105

```rust
// Component 3 (ϕ3): Vertical continuity
let phi3 = if is_cross_layout {
    -my2 // Cross-layout: prefer above (negative y means higher up)
} else {
    ry1  // ❌ BUG: Absolute position of regular element's top edge
};
```

### 4. Why This Breaks

For a title at y=950 trying to match with regular text elements:

| Regular Element | y-position | phi3 value | Result |
|----------------|------------|------------|---------|
| ID 16 (top) | y=301 | phi3 = 301 | **SMALL = BEST MATCH!** ❌ |
| ID 6 (middle) | y=950 | phi3 = 950 | LARGE = worse match |
| ID 13 (bottom) | y=1441 | phi3 = 1441 | LARGE = worst match |

**The algorithm always picks the element at the TOP of the page!**

Because phi3 uses **absolute y-coordinate**, elements with smaller y-values (higher on page) always win, regardless of where the title actually is.

### 5. What phi3 Should Do

**Intent:** Measure vertical continuity - a title should match with the text that comes AFTER it (below it) in reading order.

**Correct behavior:**
- Title at y=215 should match with text at y=301 (distance = 86)
- Title at y=950 should match with text at y=976 (distance = 26), NOT with text at y=301 (which is ABOVE it!)

## The Fix

### Mental Model

**Before (broken):**
```rust
phi3 = ry1  // "How high up is this element?"
```
This favors elements at the top of the page.

**After (fixed):**
```rust
phi3 = if regular is below title {
    ry1 - my2  // "How far below the title is this element?"
} else {
    1000.0     // "This element is ABOVE the title - big penalty"
}
```
This favors the **nearest** element below the title.

### The Code Fix

**File:** `xycut-plus-plus/src/utils.rs`
**Lines:** ~100-111

```rust
// Component 3 (ϕ3): Vertical continuity
let phi3 = if is_cross_layout {
    -my2 // Cross-layout: prefer above (negative y means higher up)
} else {
    // For titles/figures: prefer regular elements BELOW them
    // Use vertical distance, penalize if regular is above
    if ry1 >= my2 {
        ry1 - my2  // Regular below masked - distance is good
    } else {
        1000.0     // Regular above masked - heavy penalty
    }
};
```

### Why This Works

For a title at y=950 (my2=962 after including height):

| Regular Element | y-position | Comparison | phi3 value | Result |
|----------------|------------|------------|------------|---------|
| ID 16 (top) | ry1=301 | 301 < 962 | phi3 = 1000.0 | **PENALTY!** ✓ |
| ID 6 (middle) | ry1=976 | 976 >= 962 | phi3 = 14 | **BEST MATCH!** ✓ |
| ID 13 (bottom) | ry1=1467 | 1467 >= 962 | phi3 = 505 | Further = worse ✓ |

Now the title at y=950 correctly matches with the nearest text BELOW it.

## Key Lessons

### 1. **Absolute vs. Relative Metrics**
- ❌ Bad: `phi3 = ry1` (absolute position)
- ✓ Good: `phi3 = ry1 - my2` (relative distance)

### 2. **Debug Output is Critical**
Without the debug output showing "all elements matching ID 16", we wouldn't have spotted the pattern.

### 3. **Paper Implementation Nuances**
The paper says "vertical continuity" but doesn't specify the exact formula for non-cross-layout cases. We had to infer:
- Titles appear BEFORE their content
- So titles should match with text BELOW them
- Elements ABOVE should be heavily penalized

### 4. **Test with Real Data**
Unit tests wouldn't catch this - you need real PDF layouts with titles scattered throughout the page to see the bug.
