---

## Background: The Bug

The DocLayout-YOLO model was producing bounding boxes that were:
- Far too large (40-100% of page dimensions)
- Misaligned with actual content
- Sometimes producing negative coordinates

**Root cause**: We assumed the model output format was `[center_x, center_y, width, height]` (xywh), but it's actually `[x1, y1, x2, y2]` (xyxy - corner coordinates).

---

## The Mental Model

### XYWH Format (Center + Size)
```
        w (width)
    ←─────────→
    ┌───────────┐  ↑
    │           │  │
    │     ●     │  │ h (height)
    │    cx,cy  │  │
    │           │  │
    └───────────┘  ↓

Center point: (cx, cy)
Dimensions: w × h
To get corners: x1 = cx - w/2, x2 = cx + w/2
```

### XYXY Format (Corners)
```
    x1,y1
      ●───────────┐
      │           │
      │           │
      │           │
      │           │
      └───────────●
                 x2,y2

Top-left: (x1, y1)
Bottom-right: (x2, y2)
Already corners - no conversion needed!
Dimensions: w = x2 - x1, h = y2 - y1
```

## Step-by-Step Fix

### Step 1: Add Debug Output

**File**: `src/models/layout.rs`

```rust
let (shape, data) = output;  // Change from (_shape, data)
let mut bboxes = Vec::new();

```

**Why**: Before fixing anything, you need to see the actual data. This shows you:
- The tensor shape (should be [1, 300, 6])
- The actual coordinate values
- Context about image dimensions

### Step 2: Update Variable Names (Critical!)

**Current code** (WRONG - treating as xywh):
```rust
// Extract detectioj values  // Note: typo in original
let cx = data[base_idx];           // center x (normalized 0-1)
let cy = data[base_idx + 1];       // center y (normalized 0-1)
let w = data[base_idx + 2];        // width (normalized 0-1)
let h = data[base_idx + 3];        // height (normalized 0-1)
let confidence = data[base_idx + 4];
let class_id = data[base_idx + 5] as usize;
```

**New code** (CORRECT - treating as xyxy):
```rust
// Extract detection values (XYXY format in 1024×1024 pixel space)
let x1_1024 = data[base_idx];       // left edge in pixels (0-1024)
let y1_1024 = data[base_idx + 1];   // top edge in pixels (0-1024)
let x2_1024 = data[base_idx + 2];   // right edge in pixels (0-1024)
let y2_1024 = data[base_idx + 3];   // bottom edge in pixels (0-1024)
let confidence = data[base_idx + 4];
let class_id = data[base_idx + 5] as usize;
```

**Key insight**: Variable names matter! When you name them `cx/cy/w/h`, your brain thinks "center and size". When you name them `x1/y1/x2/y2`, you think "corners".

**Also update the comment** at the top of the loop:
```rust
// Output is [1, 300, 6] where each detection is [x1, y1, x2, y2, confidence, class_id]
// CRITICAL: Format is XYXY (corner coordinates), NOT XYWH (center + size)!
```

---

### Step 3: Remove the Center-to-Corner Conversion

**Old transformation** (WRONG):
```rust
// Model outputs coordinates in 1024×1024 space (including padding)
// TODO: Step 1 - Scale from 1024×1024 to original image dimensions
let scale_from_1024_x = original_width / 1024.0;
let scale_from_1024_y = original_height / 1024.0;

// TODO: Step 2 - Transform box coordinates (no offset subtraction needed)
let x1_raw = (cx - w / 2.0) * scale_from_1024_x;  // ← WRONG! Produces negatives
let y1_raw = (cy - h / 2.0) * scale_from_1024_y;
let x2_raw = (cx + w / 2.0) * scale_from_1024_x;
let y2_raw = (cy + h / 2.0) * scale_from_1024_y;

// TODO: Step 3 - Clip boxes to valid image region
let x1 = x1_raw.max(0.0);
let y1 = y1_raw.max(0.0);
let x2 = x2_raw.min(original_width);
let y2 = y2_raw.min(original_height);
```

**New transformation** (CORRECT):
```rust
// Coordinates are ALREADY corners in 1024×1024 pixel space
// Transform: 1024 padded space → original image space

// Step 1: Corners are already in 1024 space (no conversion needed)
let x1_padded = x1_1024;
let y1_padded = y1_1024;
let x2_padded = x2_1024;
let y2_padded = y2_1024;

// Step 2: Subtract padding offset to get resized space coordinates
let x1_resized = x1_padded - x_offset as f32;
let y1_resized = y1_padded - y_offset as f32;
let x2_resized = x2_padded - x_offset as f32;
let y2_resized = y2_padded - y_offset as f32;

// Step 3: Scale from resized dimensions to original dimensions
let scale_x = original_width / new_width as f32;
let scale_y = original_height / new_height as f32;

let x1_original = x1_resized * scale_x;
let y1_original = y1_resized * scale_y;
let x2_original = x2_resized * scale_x;
let y2_original = y2_resized * scale_y;

// Step 4: Clip to valid image region
let x1 = x1_original.max(0.0);
let y1 = y1_original.max(0.0);
let x2 = x2_original.min(original_width);
let y2 = y2_original.min(original_height);
```

---

## Understanding the Transformation

### Why This Works (The Math)

**Given**:
- Original image: 1268×1654 pixels
- Letterbox resizes to: 785×1024 pixels (preserves aspect ratio)
- Pads to: 1024×1024 pixels with offset (119, 0)
- Model outputs coordinates in this 1024×1024 padded space

**Example box**: `x1=188, y1=458, x2=489, y2=841`

**Step 1**: Already corners (no work)
```
x1_padded = 188, x2_padded = 489
Box width in 1024 space = 489 - 188 = 301 pixels
```

**Step 2**: Remove padding offset
```
x1_resized = 188 - 119 = 69
x2_resized = 489 - 119 = 370
Now in 785×1024 space (resized image coordinates)
```

**Step 3**: Scale to original dimensions
```
scale_x = 1268 / 785 = 1.615
x1_original = 69 × 1.615 = 111.4
x2_original = 370 × 1.615 = 597.6
Final box width = 597.6 - 111.4 = 486 pixels (38% of page)
```

**Step 4**: Clip (already in valid range, no change)

## Common Mistakes to Avoid

### Mistake 1: Forgetting to Update Comments
```rust
// WRONG - outdated comment
// Output is [cx, cy, w, h, ...]  ← This will confuse you later!

// RIGHT - accurate comment
// Output is [x1, y1, x2, y2, ...] with xyxy format
```

### Mistake 2: Mixing Old and New Variable Names
```rust
// WRONG - using old names with new interpretation
let x1 = cx - x_offset as f32;  ← You renamed cx to x1_1024!

// RIGHT - consistent naming
let x1_resized = x1_padded - x_offset as f32;
```

### Mistake 3: Scaling Before Offset Subtraction
```rust
// WRONG order
let x1_scaled = x1_1024 * scale_x;
let x1_final = x1_scaled - x_offset as f32;  ← Offset is in 1024 space, not scaled space!

// RIGHT order
let x1_resized = x1_1024 - x_offset as f32;  ← First remove padding
let x1_final = x1_resized * scale_x;         ← Then scale to original
```

## What You Learned

### Technical Skills
- **Format interpretation**: Always verify model output format, don't assume
- **Coordinate transformations**: Order matters (remove offset BEFORE scaling)
- **Debugging ML outputs**: Add logging to see raw tensor values
- **Variable naming**: Good names prevent bugs (`x1` vs `cx` conveys intent)

### Debugging Process
1. **Observe**: Boxes were wrong → added debug output
2. **Hypothesize**: Tried 7+ transformations → all failed
3. **Research**: Checked model docs → found xyxy format
4. **Verify**: Compared Python/Rust outputs → confirmed format
5. **Fix**: One-line change → working perfectly

### Key Insight
Sometimes the bug isn't in your complex transformation logic - it's in your basic assumptions about the data format. When nothing works, question your assumptions!
