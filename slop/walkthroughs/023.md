# XY-Cut++ Reading Order Debugging & Fixes

**Date**: 2025-10-31
**Component**: xycut-plus-plus library
**Goal**: Fix reading order detection bugs in multi-column academic PDFs

---

## Summary

Implemented XY-Cut++ reading order detection algorithm and debugged four critical bugs that caused incorrect element ordering in multi-column documents.

**Final Result**: Reading order is now correct with proper column detection, no column jumping, correct positioning of spanning elements, and accurate placement of section headers regardless of box width.

---

## Bug #1: Duplicate Boxes in Reading Order

### Symptom
Multiple boxes appeared in the final reading order because they shared the same `class_id`.

### Root Cause
The `BoundingBox` trait's `id()` method returned `class_id` (element type: 0=title, 1=text, etc.) instead of a truly unique identifier.

```rust
// BEFORE (WRONG)
impl xycut_plus_plus::BoundingBox for BBox {
    fn id(&self) -> usize {
        self.class_id  // All titles have class_id=0!
    }
}
```

Multiple elements of the same type (e.g., all titles, all text blocks) had identical IDs, causing duplicate/missing boxes in the output.

### Fix
**File**: `/mnt/datadrive_m2/pdf_masher/pdf-mash/src/models/layout.rs`

1. Added `unique_id` field to `BBox` struct:
```rust
pub struct BBox {
    // ... existing fields
    pub unique_id: usize, // Unique identifier for reading order
}
```

2. Assigned unique IDs during detection (line 287):
```rust
bboxes.push(BBox {
    // ... other fields
    unique_id: bboxes.len(), // Sequential unique ID
});
```

3. Updated trait implementation to use `unique_id`:
```rust
impl xycut_plus_plus::BoundingBox for BBox {
    fn id(&self) -> usize {
        self.unique_id  // Now truly unique!
    }
}
```

**Result**: ✓ Every box now has a unique identifier, eliminating duplicates

## Bug #2: Column Jumping in Cross-Modal Matching

### Symptom
Reading order jumped between columns instead of completing one column before moving to the next:

```
14. plain text at y=1419, x=97    <- left column
15. title at y=1430, x=612        <- JUMPED to right column!
16. plain text at y=1419, x=97    <- back to left column
```

### Root Cause
**File**: `/mnt/datadrive_m2/pdf_masher/xycut-plus-plus/src/core.rs:250-271`

The `merged_masked_elements` function's fallback logic (when IoU=0) only checked y-position for insertion, completely ignoring which column elements belonged to:

```rust
// BEFORE (WRONG) - Only checked y-position
for (i, regular_id) in result.iter().enumerate() {
    if let Some(regular) = regular_elements.iter().find(|e| e.id() == *regular_id) {
        let (regular_x, regular_y) = regular.center();

        // Only checks y-position, not x-position!
        if regular_y > masked_y {
            insert_pos = i;
            break;
        }
    }
}
```

This caused masked section headers from the right column (e.g., at x=612, y=1430) to be inserted between left column elements with similar y-coordinates (x=97, y=1419).

### Fix
Made cross-modal matching column-aware by checking BOTH x-position and y-position:

```rust
// AFTER (CORRECT) - Column-aware matching
let (masked_x, masked_y) = masked.center();

for (i, regular_id) in result.iter().enumerate() {
    if let Some(regular) = regular_elements.iter().find(|e| e.id() == *regular_id) {
        let (regular_x, regular_y) = regular.center();

        // Check if in same column (x within 150 pixels)
        let same_column = (regular_x - masked_x).abs() < 150.0;

        // Insert before this element if it's in same column AND below masked element
        if same_column && regular_y > masked_y {
            insert_pos = i;
            break;
        }
    }
}
```

**Result**: ✓ Masked elements now respect column boundaries (150px tolerance)

## Bug #3: Spanning Title Positioned Incorrectly

### Symptom
The main page title (spanning both columns) appeared at position 5 instead of position 1:

```
1. plain text at y=301, x=110
2. plain text at y=304, x=784
...
5. title at y=187, x=96          <- Should be #1! It's the topmost element!
```

### Root Cause
**File**: `/mnt/datadrive_m2/pdf_masher/xycut-plus-plus/src/core.rs:228`

Two compounding issues:

1. **Masked elements were processed in arbitrary order** (not sorted by position)
2. **Spanning elements failed column-aware matching**:
   - Title at y=187 spans x=96 to x=1095 (width=934)
   - Center x ≈ 595 (middle of page)
   - Doesn't match left column (x=110): `|110 - 595| = 485 > 150px`
   - Doesn't match right column (x=784): `|784 - 595| = 189 > 150px`
   - Defaults to `insert_pos = result.len()` (end of list)

### Fix Part 1: Sort Masked Elements Before Merging
Ensures topmost elements are processed first:

```rust
fn merged_masked_elements<T: BoundingBox>(
    &self,
    regular_elements: &[T],
    regular_order: &[usize],
    masked_elements: &[T],
) -> Vec<usize> {
    let mut result: Vec<usize> = regular_order.to_vec();

    // NEW: Sort masked elements by position before merging
    let mut sorted_masked: Vec<T> = masked_elements.to_vec();
    sorted_masked.sort_by(|a, b| {
        let y_diff = (a.center().1 - b.center().1).abs();
        if y_diff < self.config.same_row_tolerance {
            a.center().0.partial_cmp(&b.center().0).unwrap()
        } else {
            a.center().1.partial_cmp(&b.center().1).unwrap()
        }
    });

    // Process in top-to-bottom, left-to-right order
    for masked in &sorted_masked {
        // ... insertion logic
    }
}
```

### Fix Part 2: Handle Spanning Elements Specially
Detect wide-spanning elements (>60% page width) and insert based on y-position only:

```rust
// Check if this is a wide spanning element
let masked_bounds = masked.bounds();
let masked_width = masked_bounds.2 - masked_bounds.0;
let page_width = self.compute_page_width(regular_elements);
let is_spanning = masked_width > (page_width * 0.6);

for (i, regular_id) in result.iter().enumerate() {
    if let Some(regular) = regular_elements.iter().find(|e| e.id() == *regular_id) {
        let (regular_x, regular_y) = regular.center();

        if is_spanning {
            // Spanning element - insert based on y-position only
            if regular_y > masked_y {
                insert_pos = i;
                break;
            }
        } else {
            // Normal column-aware matching (150px tolerance)
            let same_column = (regular_x - masked_x).abs() < 150.0;
            if same_column && regular_y > masked_y {
                insert_pos = i;
                break;
            }
        }
    }
}
```

Added helper function:
```rust
fn compute_page_width<T: BoundingBox>(&self, elements: &[T]) -> f32 {
    if elements.is_empty() {
        return 0.0;
    }
    let x_min = elements.iter().map(|e| e.bounds().0).fold(f32::INFINITY, f32::min);
    let x_max = elements.iter().map(|e| e.bounds().2).fold(f32::NEG_INFINITY, f32::max);
    x_max - x_min
}
```

**Result**: ✓ Page titles and other spanning elements now inserted at correct position based on y-coordinate

## Additional Enhancement: Wide Element Masking

### Issue
Elements like abstracts that span both columns prevent column gap detection.

### Solution
**File**: `/mnt/datadrive_m2/pdf_masher/xycut-plus-plus/src/matching.rs`

Modified `partition_by_mask` to mask elements wider than 70% of page width:

```rust
pub fn partition_by_mask<T: BoundingBox>(elements: &[T], page_width: f32) -> MaskPartition<T> {
    let mut masked_elements = Vec::new();
    let mut regular_elements = Vec::new();

    for element in elements {
        let (x1, _, x2, _) = element.bounds();
        let width = x2 - x1;

        // Mask if: 1) should_mask() returns true, OR 2) element is wider than 70% of page
        let is_wide = width > (page_width * 0.7);

        if element.should_mask() || is_wide {
            masked_elements.push(element.clone());
        } else {
            regular_elements.push(element.clone());
        }
    }

    MaskPartition { masked_elements, regular_elements }
}
```

**Result**: ✓ Wide-spanning elements are masked before column detection, improving vertical cut success rate

## Bug #4: Section Headers Misplaced Due to Width Differences

### Symptom
Section headers in the left column were being inserted at the end instead of at their correct y-position:

```
Visual reading: 11 → 25 (section header) → 12
Algorithm order: 11 → 12 → ... → 25 (at the end!)
```

### Root Cause
**File**: `/mnt/datadrive_m2/pdf_masher/xycut-plus-plus/src/core.rs:305-312`

Column detection used CENTER x-coordinate distance, which fails when boxes have different widths:

```rust
// BEFORE (WRONG) - Used center distance
let same_column = (regular_x - masked_x).abs() < 150.0;
```

Example:
- Section header: width=164, center x=179 (left edge at x=97)
- Text block: width=478, center x=337 (left edge at x=98)
- Center distance: |337 - 179| = 158 > 150px ❌ FAILS!

Even though both boxes are in the same column (left edges only 1px apart), their centers are 158px apart due to width difference, exceeding the 150px tolerance.

### Fix
Use LEFT EDGE distance instead of center distance for column matching:

```rust
// AFTER (CORRECT) - Use left edge distance
let masked_left = masked_bounds.0;
let regular_left = regular.bounds().0;
let same_column = (regular_left - masked_left).abs() < 100.0;
```

**Result**: ✓ Column detection now works correctly regardless of box width variations

**Before Fix:**

```
12. plain text at y=976-1173, x=98
...
25. title at y=939-962, x=97      <- Wrong position!
```

**After Fix:**

## Key Learnings

1. **Unique identifiers matter**: Using semantic IDs (class_id) instead of unique IDs causes subtle bugs
2. **Column awareness is critical**: Multi-column layouts require x-position awareness, not just y-position
3. **Process order matters**: Sorting masked elements by position before merging ensures correct insertion order
4. **Spanning elements need special handling**: Elements that span columns don't match column-based logic
5. **Pre-masking improves column detection**: Removing wide elements before recursive cutting helps find column gaps
6. **Box geometry matters for column detection**: LEFT EDGE distance is more reliable than CENTER distance when boxes have varying widths

---

## Algorithm Overview: XY-Cut++

**Pre-mask Processing** → **Recursive XY-Cut** → **Cross-Modal Matching**

1. **Pre-mask**: Remove titles, figures, tables, and wide elements (>70% page width)
2. **Recursive Cut**:
   - Try vertical cuts first for large groups (>10 elements) to detect columns
   - Fall back to horizontal cuts for reading flow
   - Use projection histograms to find gaps
3. **Cross-Modal Matching**:
   - Sort masked elements by position (top-to-bottom, left-to-right)
   - Insert using IoU overlap when possible
   - Fall back to column-aware position matching using LEFT EDGE distance (100px tolerance)
   - Handle spanning elements (>60% page width) with y-position only

This produces correct reading order for complex multi-column academic documents.
