## Issue #1: Missing Fallback for Unmatched Elements

### Status: CRITICAL (Silent Data Loss)

### The Problem

**Location:** `xycut-plus-plus/src/core.rs`, lines ~416-419 (in `merged_masked_elements`)

Masked elements that don't find a valid insertion position are **silently dropped**:

```rust
// Current code
if let Some(position) = best_position {
    result.insert(position, masked.id());
}
// NO ELSE CLAUSE - element disappears!
```

**Impact:**
- Titles, figures, or tables could be lost from the output
- No error or warning is logged
- User has no way to detect the issue

### The Theory

**Paper's Equation 7:** The constraint L'o ⪰ l ensures we only insert after equal/higher priority elements. But what if **no valid position exists**?

**Example scenario:**
- A Vision element (priority 2)
- All existing elements are CrossLayout (priority 0) or HorizontalTitle (priority 1)
- Vision can't insert before any of them (constraint violation)
- Current code: Vision element is lost!

**Correct behavior:**
- If no valid match found, append to end (safe fallback)
- Element is preserved in output

### The Fix

**File:** `xycut-plus-plus/src/core.rs`
**Location:** In `merged_masked_elements`, around line 416

Replace:
```rust
// Insert masked element at best position
if let Some(position) = best_position {
    result.insert(position, masked.id());
}
```

With:
```rust
// Insert masked element at best position, or append if no valid match
if let Some(position) = best_position {
    result.insert(position, masked.id());  // Insert BEFORE matched element
} else {
    // No valid match found - append to end as fallback
    eprintln!(
        "⚠️  No valid insertion position for masked element {} ({}), appending to end",
        masked.id(),
        masked.semantic_label() as u8
    );
    result.push(masked.id());
}
```

**Key changes:**
1. **Added `else` clause** - Prevents silent data loss
2. **Append as fallback** - Ensures element is included
3. **Warning log** - Makes issue visible for debugging
4. **Insert at `position`** - Insert BEFORE the matched element (reading order)

### Why position (not position + 1)?

**CORRECTION:** Earlier versions of this walkthrough incorrectly stated `position + 1`.

**Reading order logic:**
- Distance metric finds the element that should come AFTER the masked element
- We want masked element to come BEFORE that element
- `insert(position, item)` inserts BEFORE position
- So we use `insert(position, item)` to insert BEFORE the matched element

**Example:**
```
result = [A, B, C, D]  (indices: 0, 1, 2, 3)
best_position = 2      (element C - should come AFTER masked)
Want: [A, B, MASKED, C, D]

result.insert(2, M)  → [A, B, M, C, D]  ✅ Correct! M comes before C
result.insert(3, M)  → [A, B, C, M, D]  ❌ Wrong! M comes after C
```

**Bug found 2025-11-16:** Using `position + 1` caused titles to appear AFTER body text. See `slop/journal/2025-11-16-xycut-insertion-position-bug.md` for details.

### Testing

Add debug output to verify:
```rust

    "Inserting {} after position {} (element {})",
    masked.id(),
    position,
    result[position]
);
```

## Issue #2: Component φ3 Not Matching Paper

### Status: CRITICAL (Algorithm Incorrectness)

### The Problem

**Location:** `xycut-plus-plus/src/utils.rs`, lines ~119-134

Current implementation of φ3 (vertical continuity) doesn't match paper's Equation 8.

**Current code:**
```rust
let phi3 = if is_cross_layout {
    if my1 > ry2 {
        my1 - ry2  // Masked below regular - penalize
    } else {
        -my2       // Masked above or overlaps
    }
} else {
    if ry1 >= my2 {
        ry1 - my1  // Regular below - baseline alignment
    } else {
        (my2 - ry1) * 10.0  // Regular above - scaled penalty
    }
};
```

**Paper's specification (Equation 8):**
- For cross-layout above regular: φ3 = **-y'2** (prefer elements higher on page)
- For regular below masked: φ3 = **y'1** (baseline alignment)

### The Theory

**Paper's intent for φ3:**

1. **Cross-layout elements** (wide spanning titles):
   - Should be placed near the **top** of the page
   - Negative y-coordinate means "higher priority for lower y values"
   - Formula: φ3 = -y'2 (where y'2 is the bottom edge of regular element)

2. **Single-column elements** (titles, figures):
   - Should align with text immediately **below** them
   - Reading flows top-to-bottom
   - Formula: φ3 = y'1 (where y'1 is the top edge of regular element)

### Current Issues

1. **Complexity:** Too many conditional branches
2. **Scaled penalties:** Not in the paper
3. **Logic mismatch:** Doesn't align with paper's simple formulas

### The Fix

**File:** `xycut-plus-plus/src/utils.rs`
**Location:** Lines ~119-134

Replace the entire phi3 calculation with:

```rust
// Component 3 (ϕ3): Vertical continuity (Equation 8)
let phi3 = if is_cross_layout {
    // Cross-layout: Prefer placement near top of page
    // Paper specifies: φ3 = -y'2 (negative y means higher up)
    -ry2
} else {
    // Single column: Baseline alignment with element below
    // Paper specifies: φ3 = y'1 (top edge of regular element)
    ry1
};
```

**Why this is simpler and correct:**

1. **Cross-layout:**
   - Uses `-ry2` (negative bottom of regular element)
   - Lower y values (top of page) = more negative = smaller distance = preferred
   - Directly from paper's Equation 8

2. **Single-column:**
   - Uses `ry1` (top of regular element)
   - Elements immediately below masked = smaller y1 = smaller distance = preferred
   - Directly from paper's Equation 8

3. **No penalties needed:**
   - Weight `w3` already controls the influence
   - Other components (φ1, φ2, φ4) handle constraints

### Expected Behavior Change

**Before (complex):**
- Cross-layout: Different behavior if masked is below vs above
- Single-column: 10x penalty if regular is above
- Many conditional branches

**After (simple):**
- Cross-layout: Consistently prefer higher placement
- Single-column: Consistently prefer baseline alignment
- Matches paper exactly

2. Document with multiple single-column sections

## Issue #3: Remaining .unwrap() Calls

### Status: HIGH (Potential Panics)

### The Problem

Several `.unwrap()` calls remain that could panic on NaN values:

**Locations:**
1. `xycut-plus-plus/src/utils.rs:136` - Sorting widths
2. `pdf-mash/src/models/layout.rs:302` - Sorting by confidence
3. `pdf-mash/src/utils/deduplication.rs:8` - Sorting by confidence

**Why this is dangerous:**

`partial_cmp` returns `None` when comparing NaN values:
```rust
let a = f32::NAN;
let b = 5.0;
a.partial_cmp(&b)  // Returns None
a.partial_cmp(&b).unwrap()  // PANIC!
```

If coordinates or confidence scores contain NaN (due to bad OCR, division by zero, etc.), the program crashes.

### The Fix

Replace all three occurrences:

#### Fix #1: utils.rs - Median width calculation

**File:** `xycut-plus-plus/src/utils.rs`
**Line:** ~136

Find:
```rust
widths.sort_by(|a, b| a.partial_cmp(b).unwrap_or(std::cmp::Ordering::Equal));
```

Already correct! ✅ (This was fixed in a previous session)

#### Fix #2: layout.rs - Sort by confidence

**File:** `pdf-mash/src/models/layout.rs`
**Line:** ~302

Find:
```rust
.partial_cmp(&other.confidence).unwrap()
```

Replace with:
```rust
.partial_cmp(&other.confidence).unwrap_or(std::cmp::Ordering::Equal)
```

#### Fix #3: deduplication.rs - Sort by confidence

**File:** `pdf-mash/src/utils/deduplication.rs`
**Line:** ~8

Find:
```rust
.partial_cmp(&b.confidence).unwrap()
```

Replace with:
```rust
.partial_cmp(&b.confidence).unwrap_or(std::cmp::Ordering::Equal)
```

### Safe Comparison Pattern

**Remember this pattern:**
```rust
// ❌ UNSAFE - panics on NaN
a.partial_cmp(&b).unwrap()

// ✅ SAFE - treats NaN as equal (stable sort)
a.partial_cmp(&b).unwrap_or(std::cmp::Ordering::Equal)
```

**Alternative:** Filter out NaN values before sorting:
```rust
items.retain(|x| x.confidence.is_finite());
items.sort_by(|a, b| a.confidence.partial_cmp(&b.confidence).unwrap());
```

## Issue #4: Histogram Bounds Not Checked

### Status: MEDIUM (Potential Panic)

### The Problem

**Location:** `xycut-plus-plus/src/histogram.rs`

Array indexing without bounds checks:
```rust
for bin in start_bin..end_bin.min(resolution) {
    histogram[bin] += 1;  // Could panic if bin >= histogram.len()
}
```

**How it could fail:**
1. Floating-point rounding errors
2. Elements outside bounding box
3. Resolution calculation edge cases

### The Fix

**File:** `xycut-plus-plus/src/histogram.rs`

Find both histogram building functions and add bounds checks.

#### Horizontal Histogram

Find:
```rust
for bin in start_bin..end_bin.min(resolution) {
    histogram[bin] += 1;
}
```

Replace with:
```rust
for bin in start_bin..end_bin.min(resolution) {
    if bin < histogram.len() {
        histogram[bin] += 1;
    }
}
```

#### Vertical Histogram

Same fix in the vertical histogram function.

### Alternative: Defensive Clamping

More defensive approach:
```rust
let start_bin = start_bin.min(histogram.len().saturating_sub(1));
let end_bin = end_bin.min(histogram.len());

for bin in start_bin..end_bin {
    histogram[bin] += 1;
}
```

### Unit Tests to Add

Create `xycut-plus-plus/src/core.rs` tests:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use crate::traits::{BoundingBox, SemanticLabel};

    // Mock BBox for testing
    #[derive(Clone)]
    struct TestBBox {
        id: usize,
        x1: f32, y1: f32, x2: f32, y2: f32,
        label: SemanticLabel,
    }

    impl BoundingBox for TestBBox {
        fn bounds(&self) -> (f32, f32, f32, f32) {
            (self.x1, self.y1, self.x2, self.y2)
        }
        fn id(&self) -> usize { self.id }
        fn semantic_label(&self) -> SemanticLabel { self.label }
        fn should_mask(&self) -> bool {
            !matches!(self.label, SemanticLabel::Regular)
        }
    }

    #[test]
    fn test_empty_input() {
        let xycut = XYCutPlusPlus::new(XYCutConfig::default());
        let result = xycut.compute_order(&[], 0.0, 0.0, 100.0, 100.0);
        assert!(result.is_empty(), "Empty input should return empty result");
    }

    #[test]
    fn test_single_element() {
        let xycut = XYCutPlusPlus::new(XYCutConfig::default());
        let elements = vec![TestBBox {
            id: 42,
            x1: 0.0, y1: 0.0, x2: 10.0, y2: 10.0,
            label: SemanticLabel::Regular,
        }];
        let result = xycut.compute_order(&elements, 0.0, 0.0, 100.0, 100.0);
        assert_eq!(result, vec![42]);
    }

    #[test]
    fn test_all_masked_elements() {
        let xycut = XYCutPlusPlus::new(XYCutConfig::default());
        let elements = vec![
            TestBBox {
                id: 1,
                x1: 0.0, y1: 0.0, x2: 100.0, y2: 10.0,
                label: SemanticLabel::HorizontalTitle,
            },
            TestBBox {
                id: 2,
                x1: 0.0, y1: 20.0, x2: 100.0, y2: 30.0,
                label: SemanticLabel::Vision,
            },
        ];
        let result = xycut.compute_order(&elements, 0.0, 0.0, 100.0, 100.0);
        assert_eq!(result.len(), 2, "Should preserve all elements");
        assert!(result.contains(&1) && result.contains(&2));
    }

    #[test]
    fn test_no_valid_insertion_position() {
        // Test case: Vision element with only CrossLayout elements
        // Vision (priority 2) cannot insert before CrossLayout (priority 0)
        let xycut = XYCutPlusPlus::new(XYCutConfig::default());
        let elements = vec![
            TestBBox {
                id: 1,
                x1: 0.0, y1: 0.0, x2: 100.0, y2: 10.0,
                label: SemanticLabel::CrossLayout,
            },
            TestBBox {
                id: 2,
                x1: 10.0, y1: 50.0, x2: 50.0, y2: 100.0,
                label: SemanticLabel::Vision,
            },
        ];
        let result = xycut.compute_order(&elements, 0.0, 0.0, 100.0, 100.0);
        assert_eq!(result.len(), 2, "Both elements should be in result");
        // Vision should be appended to end
        assert_eq!(result[1], 2, "Vision should be appended after CrossLayout");
    }
}
```
