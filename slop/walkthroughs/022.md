# Walkthrough 18: Fixing Reading Order Tall-Box Bug

## What You'll Fix

A subtle but critical bug where tall boxes in one column incorrectly group boxes in another column, causing reading order to skip around.

**Before (broken)**:
```
10. plain text at y=831-902, x=97   ← Lower box (reads first - WRONG!)
11. plain text at y=657-829, x=98   ← Higher box (reads second - WRONG!)
```

**After (fixed)**:
```
10. plain text at y=657-829, x=98   ← Higher box (reads first - CORRECT!)
11. plain text at y=831-902, x=97   ← Lower box (reads second - CORRECT!)
```

---

## The Mental Model

### How The Bug Happens

```
Left Column          Right Column
┌──────────┐
│ Box 11   │         ┌────────────┐
│ y=657    │         │            │
│   -829   │         │  Box 12    │  ← TALL! (height=246px)
└──────────┘         │  y=657     │     Overlaps BOTH
                     │    -903    │     boxes 10 and 11
┌──────────┐         │            │
│ Box 10   │         │            │
│ y=831    │         └────────────┘
│   -902   │
└──────────┘
```

**What happens**:
1. Box 11 (y=657-829) overlaps with box 12 (y=657-903) → grouped together ✓
2. Box 10 (y=831-902) overlaps with box 12 (y=657-903) → added to same group ✗
3. All three in one "row", sorted by X: 10 (x=97) → 11 (x=98) → 12 (x=613)
4. Result: Box 10 comes before box 11, even though it's lower on the page!

### Root Causes

1. **Overlap-based grouping**: Checks if Y ranges overlap, not if boxes are at similar heights
2. **Greedy, order-dependent algorithm**: Adds boxes to first matching row, creating arbitrary dependencies
3. **Tall boxes create "bridges"**: One tall box can transitively group unrelated boxes

---

## Step 1: Understand the Current (Broken) Algorithm

**File**: `src/pipeline/reading_order.rs`
**Function**: `group_into_rows()`
**Lines**: 55-89

```rust
fn group_into_rows(bboxes: &[BBox]) -> Vec<Vec<usize>> {
    let mut rows: Vec<Vec<usize>> = Vec::new();
    let overlap_threshold = 10.0;

    for (idx, bbox) in bboxes.iter().enumerate() {  // ← Order-dependent!
        let mut found_row = false;

        for row in &mut rows {
            let overlaps_with_row = row.iter().any(|&other_idx| {
                boxes_overlap_vertically(bbox, &bboxes[other_idx], overlap_threshold)
            });

            if overlaps_with_row {
                row.push(idx);    // ← Add to FIRST matching row
                found_row = true;
                break;            // ← Stop checking other rows
            }
        }

        if !found_row {
            rows.push(vec![idx]);
        }
    }
    rows
}
```

**Problems**:
- Processes boxes in detection order (arbitrary)
- Adds to first row that overlaps
- Once added, can create transitive groupings

## Step 2: Fix Part 1 - Iterative Merging (Order-Independent)

Instead of greedily adding boxes to rows, start with each box in its own row, then merge rows that should be together.

**My implementation**:

```rust
fn group_into_rows(bboxes: &[BBox]) -> Vec<Vec<usize>> {
    if bboxes.is_empty() {
        return Vec::new();
    }

    let overlap_threshold = 10.0; // pixels

    // Start with each box in its own row
    let mut rows: Vec<Vec<usize>> = (0..bboxes.len()).map(|i| vec![i]).collect();

    // Iteratively merge rows that have overlapping boxes
    let mut merged = true;
    while merged {
        merged = false;

        // Try to merge each pair of rows
        let mut i = 0;
        while i < rows.len() {
            let mut j = i + 1;
            while j < rows.len() {
                // Check if any box in row i overlaps with any box in row j
                let should_merge = rows[i].iter().any(|&idx_i| {
                    rows[j].iter().any(|&idx_j| {
                        boxes_overlap_vertically(&bboxes[idx_i], &bboxes[idx_j], overlap_threshold)
                    })
                });

                if should_merge {
                    // Merge row j into row i
                    let row_j = rows.remove(j);
                    rows[i].extend(row_j);
                    merged = true;
                    // Don't increment j - we removed an element
                } else {
                    j += 1;
                }
            }
            i += 1;
        }
    }

    rows
}
```

**Pattern Spotted**: The `.any()` nested iterator checks "does ANY box in row i overlap with ANY box in row j?"

## Step 3: Fix Part 2 - Y-Center Proximity

The real fix: use Y-center distance instead of overlap.

**Replace `boxes_overlap_vertically` function** (lines 47-53):

```rust
// OLD:
fn boxes_overlap_vertically(a: &BBox, b: &BBox, threshold: f32) -> bool {

    let overlap_height = (overlap_end - overlap_start).max(0.0);
    overlap_height >= threshold
}
```

**My implementation**:

```rust
// Helper function to check if two boxes are in the same row
// Uses Y-center proximity instead of overlap to avoid tall boxes creating false groupings
fn boxes_in_same_row(a: &BBox, b: &BBox, threshold: f32) -> bool {
    let center_a = (a.y1 + a.y2) / 2.0;
    let center_b = (b.y1 + b.y2) / 2.0;
    let distance = (center_a - center_b).abs();
    distance < threshold
}
```

## Step 4: Understanding Why This Works

Let's trace through boxes 10, 11, 12:

### Old Algorithm (Overlap):
```
Box 10: y=831-902 → center = 866.5
Box 11: y=657-829 → center = 743
Box 12: y=657-903 → center = 780

Overlap check:
- Box 11 vs Box 12: ranges [657-829] and [657-903] overlap by 172px ✓
- Box 10 vs Box 12: ranges [831-902] and [657-903] overlap by 71px ✓

Result: All three in same row → sorted by X: 10 (x=97), 11 (x=98), 12 (x=613)
```

### New Algorithm (Y-Center):
```
Box 10: center = 866.5
Box 11: center = 743
Box 12: center = 780

Distance check (threshold=50px):
- Box 11 vs Box 12: |743 - 780| = 37 < 50 ✓ → same row
- Box 10 vs Box 12: |866.5 - 780| = 86.5 > 50 ✗ → different rows

Result:
- Row A: [Box 11, Box 12] → min_y = 657
- Row B: [Box 10] → min_y = 831

Sorted: Row A (657) comes before Row B (831) ✓
Within Row A: Box 11 (x=98) before Box 12 (x=613) ✓
```

**Key insight**: Y-center prevents tall boxes from bridging unrelated boxes.

## What You Learned

### Algorithm Design
- **Order independence**: Start-with-all, merge-down vs. start-empty, add-up
- **Geometric robustness**: Centers more stable than edge-based measures
- **Transitive relationships**: How indirect connections affect grouping

### Debugging Techniques
- **Print actual coordinates**: Don't guess - see the data
- **Trace through examples**: Follow algorithm step-by-step with real boxes
- **Visual verification**: Images reveal what terminal output can't

### Document Understanding
- **Multi-column layouts**: Columns can have overlapping Y ranges
- **Box height matters**: Tall elements create unexpected interactions
- **Reading flow**: Humans read top-to-bottom within columns, not just globally

---

## Performance Note

The iterative merging is O(n² × m) where:
- n = number of rows
- m = average boxes per row

For typical documents (20-30 boxes per page), this is fine. For huge documents (100s of boxes), consider union-find for O(n log n).
