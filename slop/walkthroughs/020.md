# Walkthrough 16: Implementing Reading Order for Document Layout

## What You'll Build

A reading order algorithm that transforms this chaos:
```
[Box 5: "Introduction" at y=400]
[Box 1: "The Title" at y=100]
[Box 3: "Figure 1" at y=250]
[Box 2: "Abstract" at y=150]

```

Into logical reading sequence:
```
[Box 1: "The Title" at y=100]      ← Read first
[Box 2: "Abstract" at y=150]       ← Then this
[Box 3: "Figure 1" at y=250]       ← Then figure
[Box 4: caption at y=300]          ← Then caption
[Box 5: "Introduction" at y=400]   ← Finally this
```

---

## The Mental Model

### How Humans Read Documents

When you open a paper, your eyes follow this pattern:

```
┌─────────────────────────────┐
│  [1] TITLE (top)            │  ← Always first
├─────────────────────────────┤
│  [2] Authors (below title)  │  ← Then metadata
├─────────────────────────────┤
│  [3] Abstract               │  ← Then abstract
│  spanning multiple lines    │
├──────────────┬──────────────┤
│ [4] Intro    │ [5] Column 2 │  ← Top-to-bottom,
│ text here    │ continues    │     left-to-right
├──────────────┴──────────────┤
│  [6] Figure with caption    │  ← Figures inline
└─────────────────────────────┘
```

**Key insight**: Reading order = **vertical position** (Y) first, **horizontal position** (X) second.

---

## The Algorithm: Row-Based Sorting

### Core Concept: "Visual Rows"

Boxes that **overlap vertically** belong to the same "row":

```
Box A: y1=100, y2=150  ─────┐
Box B: y1=120, y2=180  ──┐  │  ← These overlap (120 is between 100-150)
                         │  │    They're in the SAME row
Box C: y1=200, y2=250  ──┼──┘
                         └─────  Box B and C also overlap

Box D: y1=300, y2=350  ─────── Different row (no overlap with C)
```

**Within each row**: Sort left-to-right by X position.

## Step 3: Implement Vertical Overlap Detection

This is the foundation - determining if two boxes are in the same "row".

**Pattern Recognition:**

Two boxes overlap vertically if their Y ranges intersect:

```
Box A: [y1=100 ──────── y2=150]
Box B:          [y1=120 ──────── y2=180]
                 ↑
           Overlap exists! Both contain y=120-150
```

**Mathematical check:**
- Box A is in range [a.y1, a.y2]
- Box B is in range [b.y1, b.y2]
- They overlap if: `a.y1 < b.y2 AND b.y1 < a.y2`

**Let's code:**

```rust

    // HINT: Use max/min to find the overlapping region
    // HINT: Overlap height = min(a.y2, b.y2) - max(a.y1, b.y1)
    // TODO: Return true if overlap is at least `threshold` pixels
}
```

**Think about it**: Why might we need a threshold? What if boxes barely touch by 1 pixel?

<details>
<summary>Answer</summary>

A threshold prevents treating barely-touching boxes as the same row. If box A is at y=100-150 and box B is at y=150-200, they technically "touch" at y=150, but they're different rows. A threshold like 10-20 pixels filters this out.

</details>

**My implementation:**

```rust
fn boxes_overlap_vertically(a: &BBox, b: &BBox, threshold: f32) -> bool {
    let overlap_start = a.y1.max(b.y1);
    let overlap_end = a.y2.min(b.y2);
    let overlap_height = (overlap_end - overlap_start).max(0.0);

    overlap_height >= threshold
}
```

## Step 4: Group Boxes Into Rows

Now we build "rows" by grouping overlapping boxes.

**Algorithm**:
1. Start with first box → create first row
2. For each remaining box:

**My implementation:**

```rust
fn group_into_rows(bboxes: &[BBox]) -> Vec<Vec<usize>> {
    if bboxes.is_empty() {
        return Vec::new();
    }

    let mut rows: Vec<Vec<usize>> = Vec::new();
    let overlap_threshold = 10.0; // pixels

    for (idx, bbox) in bboxes.iter().enumerate() {
        let mut found_row = false;

        // Check each existing row
        for row in &mut rows {
            // Does this box overlap with ANY box in this row?
            let overlaps_with_row = row.iter().any(|&other_idx| {
                boxes_overlap_vertically(bbox, &bboxes[other_idx], overlap_threshold)
            });

            if overlaps_with_row {
                row.push(idx);
                found_row = true;
                break;
            }
        }

        // No overlap found - create new row
        if !found_row {
            rows.push(vec![idx]);
        }
    }

    rows
}
```

**Pattern Spotted**: The `.any()` iterator method is perfect for "does this match ANY item?" checks.

## Step 5: Sort Rows and Boxes

Now we have rows - but they're in random order, and boxes within rows are unordered.

**What we need:**
1. Sort rows top-to-bottom (by minimum Y position)
2. Within each row, sort boxes left-to-right (by X position)

```rust
pub fn sort_reading_order(bboxes: &mut [BBox]) {
    if bboxes.is_empty() {
        return;
    }

    // Step 1: Group into rows
    let mut rows = group_into_rows(bboxes);

    // Step 2: Sort rows top-to-bottom (by minimum Y of boxes in row)
    rows.sort_by(|row_a, row_b| {
        let min_y_a = row_a.iter().map(|&idx| bboxes[idx].y1).fold(f32::INFINITY, f32::min);
        let min_y_b = row_b.iter().map(|&idx| bboxes[idx].y1).fold(f32::INFINITY, f32::min);
        min_y_a.partial_cmp(&min_y_b).unwrap()
    });

    // Step 3: Within each row, sort left-to-right
    for row in &mut rows {
        row.sort_by(|&a, &b| {
            bboxes[a].x1.partial_cmp(&bboxes[b].x1).unwrap()
        });
    }

    // Step 4: Flatten into sorted order
    let sorted_indices: Vec<usize> = rows.into_iter().flatten().collect();

    // Step 5: Reorder original array using sorted indices
    let original = bboxes.to_vec();
    for (new_pos, &old_pos) in sorted_indices.iter().enumerate() {
        bboxes[new_pos] = original[old_pos].clone();
    }
}
```

**🧠 Mental Model**: We're building a "permutation" - a mapping from old positions to new positions. Then applying that permutation to reorder the array.

## Common Mistakes

### Mistake 1: Forgetting to Sort Rows
```rust
// WRONG - rows are in random order
let rows = group_into_rows(bboxes);
for row in rows {
    // Process...
}

// RIGHT - sort rows first
let mut rows = group_into_rows(bboxes);
rows.sort_by(|a, b| { /* by Y position */ });
```

### Mistake 2: Comparing Floats with ==
```rust
// WRONG - float equality is unreliable
if a.y1 == b.y1 { ... }

// RIGHT - use partial_cmp
a.y1.partial_cmp(&b.y1).unwrap()
```

### Mistake 3: Not Handling Empty Bboxes
```rust
// WRONG - crashes on empty input
let min_y = bboxes[0].y1;

// RIGHT - early return
if bboxes.is_empty() {
    return;
}
```

## What You Learned

### Algorithm Design
- **Spatial grouping**: Using overlap detection to find relationships
- **Multi-level sorting**: Sort by primary key (Y), then secondary (X)
- **Array reordering**: Building permutations and applying them

### Rust Patterns
- **Iterator methods**: `.any()`, `.map()`, `.fold()` for elegant checks
- **Partial ordering**: Using `partial_cmp` for floats
- **Slice mutations**: Reordering in-place efficiently

### Document Understanding
- Reading order isn't just "top to bottom" - it's rows, then columns
- Small overlaps (1-2 pixels) shouldn't group boxes into same row
- Special elements (titles, captions) need positional context
