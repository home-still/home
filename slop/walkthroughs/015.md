# Fixing CTC Decoding Bug

**Session Goal:** Debug and fix garbled OCR output caused by incorrect blank token position.

---

## What's Happening

You ran OCR and got output like this:
```
0T0f0y0v0b0m0j0u0z00p0u00x0p0n0f0o00x0v0u0i00b0o0p0s0f0y020b
```

Instead of:
```
Sexuality of women with anorexia
```

Notice the pattern: `0T0f0y0v0b0m` - there's a "0" between every letter.

---

## Mental Model: CTC Blank Tokens

CTC models use a special "blank" token to indicate "no character here". The model outputs a probability distribution at each timestep:

```
Time:    0     1     2     3     4     5
         ↓     ↓     ↓     ↓     ↓     ↓
Probs:  [B]   [H]   [B]   [E]   [L]   [L]   [O]
         ↓     ↓     ↓     ↓     ↓     ↓     ↓
Decode:  -     H     -     E     L     -     O  → "HELLO"
```

Where `[B]` is blank, `-` means skip it.

**The Bug:** Your code thinks blank is at position 95 (after all characters). But PaddleOCR puts blank at position 0. So:
- Position 0 in char_dict = "0" (the digit)
- Position 0 in model output = BLANK

Your code is treating blanks as the character "0".

**Look at your char_dict** (`models/ppocr_keys_v1.txt`):

- Index 1: "0" (maps to char_dict[0])
- Index 2: "1" (maps to char_dict[1])
- ...
- Index 95: " " (maps to char_dict[94])

### Change 2: Character Mapping

When `max_idx` is NOT blank, you currently do:
```rust
text.push_str(&self.char_dict[max_idx]);
```

**The problem:** If the model says index 1 (which is "0"), you're looking up char_dict[1] (which is "1"). Off by one!

**The fix:**
- Model index 0 = BLANK
- Model index 1 = char_dict[0] = "0"
- Model index 2 = char_dict[1] = "1"
- ...

So: `char_dict_index = max_idx - 1`

**Your Move:** Replace the line where you push to text. Calculate `char_idx` first, then push `char_dict[char_idx]`.

Make sure you bounds-check: `if char_idx < self.char_dict.len()`

### Change 3: Better Last Index Tracking

Your current code uses:
```rust
let mut last_idx = blank_idx;
```

But you're comparing model indices directly. Change to `Option<usize>` for clarity:
```rust

```

Then when comparing:
```rust

}
```

And when updating:

## Pattern You're Learning

**Off-by-one errors in ML models** are common when:
1. Blank tokens exist
2. Model uses 1-based indexing but your arrays are 0-based
3. Documentation doesn't clearly specify token positions

Always verify by:
1. Checking first few outputs manually
2. Looking at actual model inference results
3. Comparing with reference implementations

---

## Mental Model Check

**Why does blank need to be index 0?**

Because CTC was designed to handle variable-length sequences. The model might output:
```
BLANK BLANK H H E BLANK L L O BLANK
```

The decoder collapses repeated chars and removes blanks:
```
HELLO
```

Putting blank at index 0 is just a convention PaddleOCR chose. Other models might use the last index. Always check!
