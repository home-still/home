## Issue #1: Multi-Stage Semantic Processing (CRITICAL)

### The Problem

**File:** `xycut-plus-plus/src/core.rs`, lines ~329-361

The hierarchical mask mechanism is the **core innovation** of XY-Cut++. The algorithm should process masked elements in **priority groups**:

1. All CrossLayout elements first
2. Then all Title elements
3. Then all Vision elements
4. Finally Regular elements

**Current (Broken):**
```rust
// Lines 329-361
sort_masked.sort_by(|a, b| {
    // Only sorting by y-position, ignoring semantic priority!
    let y_diff = (a.center().1 - b.center().1).abs();
    if y_diff < self.config.same_row_tolerance {
        a.center().0.partial_cmp(&b.center().0).unwrap()
    } else {
        a.center().1.partial_cmp(&b.center().1).unwrap()
    }
});
```

This processes all masked elements in **reading order**, not **priority order**.

### Why This Breaks the Algorithm

**Paper's Algorithm 1 (lines 21-25):**
```
For each semantic label l in Lorder do:
    For each masked element Bp with label l:
        Find best insertion position in result
        Insert Bp into result
```

The paper processes **by priority**, so:
- CrossLayout titles spanning columns get inserted first
- Then section titles within columns
- Then figures/tables
- Finally regular text

**Without this:** A regular text element might get placed before a high-priority title, breaking reading order.

### The Fix

Replace lines 329-361 with:

```rust
// Group masked elements by semantic priority (Equation 7)
let mut priority_groups: Vec<Vec<T>> = vec![Vec::new(); 4];

for element in masked_elements {
    let priority = Self::label_priority(element.semantic_label()) as usize;
    if priority < 4 {
        priority_groups[priority].push(element.clone());
    }
}

// Process each priority group in order (CrossLayout → Title → Vision → Regular)
for mut group in priority_groups {
    // Within each priority group, sort by reading order (y, then x)
    group.sort_by(|a, b| {
        let y_diff = (a.center().1 - b.center().1).abs();
        if y_diff < self.config.same_row_tolerance {
            a.center().0.partial_cmp(&b.center().0)
                .unwrap_or(std::cmp::Ordering::Equal)
        } else {
            a.center().1.partial_cmp(&b.center().1)
                .unwrap_or(std::cmp::Ordering::Equal)
        }
    });

    // Process each element in this priority group
    for masked in &group {
        // ... existing matching and insertion logic ...
    }
}
```

### Expected Behavior Change

**Before (broken):**
```
Processing order: Title(y=100), Figure(y=200), Title(y=300), Text(y=400)
Result: Title, Figure, Title, Text (mixed priorities)
```

**After (correct):**
```
Processing order:
  Priority 0 (CrossLayout): None
  Priority 1 (Titles): Title(y=100), Title(y=300)
  Priority 2 (Vision): Figure(y=200)
  Priority 3 (Regular): Text(y=400)
Result: Title, Title, Figure, Text (grouped by priority, sorted within groups)
```

## Issue #2: Insertion Position Logic Bug (CRITICAL)

### The Problem

**File:** `xycut-plus-plus/src/core.rs`, lines 369-394

The code iterates over `regular_order` to find matches, but inserts into `result`. These arrays have **different sizes** as masked elements are inserted.

**Current (Broken):**
```rust
// Line 369: Iterate over regular_order
for regular_id in regular_order.iter() {
    if let Some(regular) = regular_elements.iter().find(|e| e.id() == *regular_id) {
        // Calculate distance, find best match
        let distance = compute_distance_with_early_exit(masked, regular, best_distance);
        if distance < best_distance {
            best_distance = distance;
            best_regular_id = Some(*regular_id);
        }
    }
}

// Line 390: Find position in result (DIFFERENT ARRAY!)
if let Some(matched_id) = best_regular_id {
    if let Some(position) = result.iter().position(|&id| id == matched_id) {
        result.insert(position, masked.id());
    }
}
```

### Why This Breaks

**Example:**
```
Initial state:
  regular_order = [1, 2, 3, 4]  (static, never changes)
  result = [1, 2, 3, 4]         (grows as masked elements inserted)

After inserting first masked element (ID 10):
  result = [1, 10, 2, 3, 4]

Second masked element wants to match with ID 3:
  - Search finds ID 3 in regular_order at position 2
  - But in result, ID 3 is now at position 3!
  - Insertion happens at wrong position
```

### The Fix

Search and insert in the **same array** (`result`):

```rust
// Search in result, not regular_order
for (idx, &elem_id) in result.iter().enumerate() {
    // Find the element (could be regular or previously inserted masked)
    let candidate = regular_elements.iter()
        .find(|e| e.id() == elem_id)
        .cloned()
        .or_else(|| {
            // Also check masked elements that were already inserted
            masked_elements.iter()
                .find(|e| e.id() == elem_id)
                .cloned()
        });

    if let Some(candidate) = candidate {
        // Check priority constraint
        let candidate_priority = Self::label_priority(candidate.semantic_label());
        if candidate_priority < masked_priority {
            continue;  // Skip higher priority elements
        }

        // Calculate distance
        let distance = compute_distance_with_early_exit(masked, &candidate, best_distance);
        if distance < best_distance {
            best_distance = distance;
            best_position = idx;  // Track position in result, not ID
        }
    }
}

// Insert at tracked position
if best_distance < f32::INFINITY {
    result.insert(best_position, masked.id());
}
```

### Expected Behavior Change

**Before (broken):** Masked elements after the first one often inserted at wrong positions due to index drift.

**After (correct):** All masked elements inserted at correct positions based on distance metric.

## Issue #3: Incomplete Priority Constraint (CRITICAL)

### The Problem

**File:** `xycut-plus-plus/src/core.rs`, lines 372-377

The L'o ⪰ l constraint (Equation 7) checks if candidates have **equal or lower priority** than the masked element. But it only checks against **regular elements**, not previously placed **masked elements**.

**Current (Broken):**
```rust
for regular_id in regular_order.iter() {
    if let Some(regular) = regular_elements.iter().find(|e| e.id() == *regular_id) {
        let regular_priority = Self::label_priority(regular.semantic_label());
        if regular_priority < masked_priority {
            continue;  // Only checks regular elements!
        }
        // ...
    }
}
```

### Why This Breaks

**Example:**
```
Processing order: CrossLayout(priority 0), then Title(priority 1)

1. CrossLayout gets inserted into result
2. Title tries to find insertion point
3. Title SHOULD NOT match with CrossLayout (priority 0 < priority 1)
4. But current code only checks regular elements, so Title might incorrectly match with CrossLayout
```

### The Fix

Check priority against **ALL elements in result** (both regular and masked):

```rust
// Already fixed in Issue #2's solution above
// When iterating result, we check candidate_priority < masked_priority
// This works for both regular AND masked elements
```

## Issue #4: Unsafe `.unwrap()` Calls (SAFETY)

### The Problem

**Locations:**
- `src/core.rs`: Lines ~280, 284, 344, 345, 347, 355, 356, 357
- `src/utils.rs`: Line ~105

`.unwrap()` on `partial_cmp` for `f32` values panics if comparing NaN.

**Current (Unsafe):**
```rust
a.center().0.partial_cmp(&b.center().0).unwrap()
```

If either coordinate is NaN, this **panics** the entire program.

### The Fix

**Option 1: Default to Equal**
```rust
a.center().0.partial_cmp(&b.center().0)
    .unwrap_or(std::cmp::Ordering::Equal)
```

**Option 2: Use total_cmp (Rust 1.62+)**
```rust
a.center().0.total_cmp(&b.center().0)
```

`total_cmp` handles NaN gracefully (treats as greater than all other values).

### Apply Everywhere

Search for all instances:
```bash
grep -n "\.unwrap()" src/core.rs src/utils.rs
```

Replace each one with `.unwrap_or(std::cmp::Ordering::Equal)` or `.total_cmp()`.

## Issue #5: Missing Input Validation (SAFETY)

### The Problem

**File:** `xycut-plus-plus/src/core.rs`, `compute_order` function

No validation of inputs:
- Empty element lists
- Invalid coordinates (NaN, infinity)
- Invalid page dimensions

### The Fix

Add validation at start of `compute_order`:

```rust
pub fn compute_order(
    &self,
    elements: &[T],
    page_width: f32,
    page_height: f32,
) -> Vec<usize> {
    // Validate inputs
    if elements.is_empty() {
        return Vec::new();
    }

    // Validate page dimensions
    if !page_width.is_finite() || !page_height.is_finite() ||
       page_width <= 0.0 || page_height <= 0.0 {
        eprintln!("Warning: Invalid page dimensions ({}, {}), using defaults",
            page_width, page_height);
        // Could return empty or use sensible defaults
        return Vec::new();
    }

    // Validate element coordinates
    let valid_elements: Vec<T> = elements.iter()
        .filter(|e| {
            let (x1, y1, x2, y2) = e.bounds();
            x1.is_finite() && y1.is_finite() &&
            x2.is_finite() && y2.is_finite() &&
            x2 > x1 && y2 > y1
        })
        .cloned()
        .collect();

    if valid_elements.len() < elements.len() {
        eprintln!("Warning: Filtered {} invalid elements",
            elements.len() - valid_elements.len());
    }

    // Continue with valid_elements instead of elements
    // ...
}
```

## Issue #6: Missing Edge Weights (FEATURE)

### The Problem

**File:** `xycut-plus-plus/src/utils.rs`, `compute_distance_with_early_exit`

The paper's Equation 10 includes **wedge** (edge weights) that vary based on element orientation, but implementation uses fixed weights.

### The Fix

Detect orientation and apply appropriate weights:

```rust
pub fn compute_distance_with_early_exit<T: BoundingBox>(...) -> f32 {
    let (mx1, my1, mx2, my2) = masked.bounds();
    let (rx1, ry1, rx2, ry2) = regular.bounds();

    // Calculate dimensions
    let mw = mx2 - mx1;
    let mh = my2 - my1;

    // Determine orientation
    let is_horizontal = mw > mh;

    // Get semantic label
    let label = masked.semantic_label();
    let is_cross_layout = matches!(label, SemanticLabel::CrossLayout);

    // ... component calculations ...

    // Equation 10: Semantic AND orientation-specific multipliers
    let (mult_w1, mult_w2, mult_w3, mult_w4) = match (label, is_horizontal) {
        (SemanticLabel::CrossLayout, _) => (1.0, 1.0, 0.1, 1.0),
        (SemanticLabel::HorizontalTitle, true) => (1.0, 0.1, 0.1, 1.0),
        (SemanticLabel::HorizontalTitle, false) => (1.0, 0.5, 0.5, 1.0),  // Vertical orientation
        (SemanticLabel::VerticalTitle, false) => (0.2, 0.1, 1.0, 1.0),
        (SemanticLabel::VerticalTitle, true) => (0.5, 0.5, 1.0, 1.0),     // Horizontal orientation
        (SemanticLabel::Vision, _) => (1.0, 1.0, 1.0, 0.1),
        (SemanticLabel::Regular, _) => (1.0, 1.0, 1.0, 0.1),
    };

}
```

## Issue #7: Component ϕ3 Implementation (FEATURE)

### The Problem

**File:** `xycut-plus-plus/src/utils.rs`, lines ~100-111

Current implementation of ϕ3 (vertical continuity) doesn't match paper's baseline alignment concept.

**Current:**
```rust
let phi3 = if is_cross_layout {
    -my2
} else {
    if ry1 >= my2 {
        ry1 - my2  // Distance below
    } else {
        1000.0     // Penalty if above
    }
};
```

**Paper's Intent (from Equation 8):**
- ϕ3 should measure **vertical continuity** based on baseline alignment
- For cross-layout: Prefer elements above (-y2 to favor top placement)
- For single column: Use y'1 (adjusted y-coordinate for baseline alignment)

### The Fix

Closer to paper's formula:

```rust
let phi3 = if is_cross_layout {
    // Cross-layout: prefer above the current position
    if my1 > ry2 {
        -ry2  // Masked is below regular - negative favors earlier placement
    } else {
        ry1   // Default baseline alignment
    }
} else {
    // Single column: baseline alignment
    // Prefer elements that come after (below) the masked element
    if ry1 >= my2 {
        ry1 - my2  // Small distance = good vertical flow
    } else {
        my1 - ry2  // If regular is above, use different metric
    }
};
```

**Note:** Paper's exact ϕ3 formula isn't fully specified. Current implementation is a reasonable interpretation.
