# Bounding Box Deduplication Walkthrough

## The Problem

**Symptom:** Duplicate or overlapping bounding boxes in layout detection output, causing:
- Elements with identical coordinates appearing multiple times in reading order
- Overlapping visualizations in output images
- Redundant text extraction

**Example from your PDF:**
```
3. title at y=304-327, x=784
4. plain text at y=304-327, x=784  ← Same coordinates!
```

Both elements occupy the exact same space - likely the layout detector classified the same region as both "title" and "plain text".

## Why This Happens

### 1. **DocLayout-YOLO Context**
- Based on YOLOv10, which uses "NMS-free" training
- YOLOv10 handles **same-class duplicates** internally (multiple "title" predictions for same region)
- But **cross-class duplicates** can still occur (same region detected as "title" AND "plain text")

### 2. **Root Causes**
- **Ambiguous regions**: Model uncertain about classification → outputs multiple labels
- **Training data artifacts**: Overlapping annotations in training set
- **Confidence thresholds**: Both predictions exceed confidence threshold
- **Multi-scale detection**: Same region detected at different scales with different labels

## State-of-the-Art Solutions (2024-2025)

### 1. **Standard NMS (Non-Maximum Suppression)**
**How it works:**
```python
1. Sort boxes by confidence score (highest first)
2. For each box:
   - Calculate IoU with all remaining boxes
   - Remove boxes with IoU > threshold
3. Return kept boxes
```

**Pros:**
- Simple, fast
- Industry standard
- Effective for same-class duplicates

**Cons:**
- Hard threshold - completely removes boxes
- Doesn't handle cross-class duplicates well
- May remove legitimate overlaps (e.g., caption near figure)

**IoU Thresholds:**
- 0.5-0.7: Standard object detection
- 0.9-0.95: Near-exact duplicates only

### 2. **Soft NMS**
**How it works:**
- Instead of removing overlapping boxes, **decay their confidence scores**
- Uses linear or Gaussian decay functions
- Boxes with very low confidence can be filtered later

**Pros:**
- Handles crowded scenes better
- Doesn't completely remove boxes

**Cons:**
- More complex
- Requires tuning decay parameters
- Overkill for document layout (pages aren't "crowded" like natural images)

### 3. **Class-Aware NMS (RECOMMENDED for Document Layout)**
**How it works:**
```python
1. Group boxes by class label
2. Apply standard NMS within each class
3. For cross-class overlaps, apply domain-specific rules
```

**Domain rules for documents:**
- If IoU > 0.9 between different classes:
  - Prefer semantic elements (title, caption) over generic (plain text)
  - Prefer higher confidence
  - Or: merge into single box with primary label

**Pros:**
- Preserves legitimate cross-class overlaps (caption above figure)
- Removes true duplicates
- Domain-aware decisions

**Cons:**
- Requires domain knowledge
- More complex logic

### 4. **YOLOv10's NMS-Free Approach**
**How it works:**
- Uses "consistent dual assignments" during training
- One-to-one assignment between predictions and ground truth
- Eliminates need for NMS during inference

**Note:** DocLayout-YOLO inherits this, so same-class duplicates should be rare. Your duplicates are **cross-class**, which NMS-free training doesn't handle.

## Recommended Solution for pdf-mash

### **Two-Stage Deduplication**

#### Stage 1: Exact Duplicate Removal (IoU ≥ 0.95)
For nearly identical boxes:
- Keep box with higher confidence score
- Or: Keep box with more specific semantic label (title > plain text)

#### Stage 2: Class-Aware NMS (IoU ≥ 0.7)
For significant overlaps:
- Within same class: Keep highest confidence
- Across classes: Apply priority rules

### **Class Priority Rules**
```rust
Priority (high to low):
1. Title, Caption, Section-Header (semantic markers)
2. Figure, Table, Equation (content blocks)
3. Plain Text, Paragraph (generic text)
4. Abandon, Footnote (auxiliary)
```

**Logic:** If two boxes overlap significantly and have different classes, keep the more **semantically specific** one.

### Algorithm Pseudocode

```rust
fn deduplicate_boxes(boxes: Vec<BBox>) -> Vec<BBox> {
    let mut result = Vec::new();
    let mut sorted = boxes.clone();

    // Sort by confidence (highest first)
    sorted.sort_by(|a, b| b.confidence().partial_cmp(&a.confidence()).unwrap());

    for candidate in sorted {
        let mut should_keep = true;

        for kept in &result {
            let iou = candidate.iou(kept);

            // Exact duplicates: skip
            if iou >= 0.95 {
                should_keep = false;
                break;
            }

            // Significant overlap with different class
            if iou >= 0.7 && candidate.class_name != kept.class_name {
                // Apply priority rules
                if class_priority(&candidate.class_name) <= class_priority(&kept.class_name) {
                    should_keep = false;
                    break;
                }
            }
        }

        if should_keep {
            result.push(candidate);
        }
    }

    result
}

fn class_priority(class: &str) -> u8 {
    match class {
        "title" | "caption" | "section_header" => 1,
        "figure" | "table" | "equation" => 2,
        "plain text" | "paragraph" => 3,
        "abandon" | "footnote" => 4,
        _ => 5,
    }
}
```

## Key Decisions

### 1. **IoU Thresholds**
- **0.95+ (exact duplicates)**: Always remove regardless of class
- **0.7-0.95 (significant overlap)**: Apply class priority rules
- **<0.7**: Keep both (legitimate overlap, e.g., caption near figure)

### 2. **Confidence vs. Class Priority**
When both are high confidence:
- **Same class**: Keep higher confidence (standard NMS)
- **Different classes**: Use class priority rules (domain knowledge)

### 3. **Conservative vs. Aggressive**
- **Conservative (IoU 0.9+)**: Only remove near-exact duplicates
- **Aggressive (IoU 0.7+)**: Remove significant overlaps

**Recommendation:** Start conservative (0.9), tune down if needed.
