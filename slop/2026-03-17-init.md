# home-still: complete download and ingestion strategy for a 3 TB academic research engine

**The entire OpenAlex–CORE–PMC pipeline fits inside 3 TB, but only with aggressive field trimming and tiered prioritization.** A realistic build ingests **~160M abstract-level vectors from OpenAlex as the metadata backbone**, supplements with **~51M full-text chunks from PMC OA** and a filtered subset of CORE, landing at roughly **250–350M total vectors** in Qdrant. Scalar quantization with mmap-backed originals keeps search performant on 64 GB RAM; on 32 GB, product quantization or reduced HNSW parameters become necessary. Embedding 250M chunks on an RTX 3090 plus three MacBooks takes approximately **2–4 days** using ONNX Runtime with FP16 inference.

This strategy document covers every step from `aws s3 sync` to Qdrant collection creation, with exact byte-level storage math, chunking recommendations validated by 2024–2025 benchmarks, and a clear priority ordering for what to cut when disk runs tight.

---

## 1. Storage budget breakdown: what fits in 3 TB

The binding constraint is **Qdrant's on-disk footprint**, which stores original vectors (for rescoring), quantized vectors, the HNSW graph, and payload data simultaneously. Every vector costs roughly **4.2 KB on disk** with scalar quantization enabled.

### Per-vector disk cost in Qdrant (768-dim)

| Component | Bytes per vector | Notes |
|-----------|-----------------|-------|
| Original vector (float32) | 768 × 4 = **3,072** | Stored on mmap for rescoring |
| Scalar quantized copy (int8) | 768 × 1 = **768** | Kept in RAM or mmap for fast search |
| HNSW graph links (m=16) | 2 × 16 × 4 = **128** | ~2m bidirectional links per node |
| Payload + point metadata | **~200** | DOI, title, year, topics, chunk metadata |
| **Total per vector** | **~4,168** | **≈ 4.2 KB** |

### Source data disk requirements

| Source | Compressed download | Processed/trimmed | Notes |
|--------|-------------------|-------------------|-------|
| **OpenAlex** | 330 GB | ~200 GB | Stream-process, discard decompressed 1.6 TB original |
| **PMC OA** (all license groups, XML) | ~60 GB | ~40 GB | JATS XML is compact |
| **CORE** (full text only) | ~150 GB (subset) | ~100 GB | Registration required; only grab full-text batches |
| **Working buffer** | ~100 GB | — | For intermediate processing, WAL, temp files |
| **Source subtotal** | ~640 GB | ~440 GB | |

### Vector storage by ingestion tier

The number of embeddable chunks drives the entire budget. OpenAlex provides **~160M papers with reconstructable abstracts** (the `abstract_inverted_index` field covers roughly 55–60% of the 288M works in the core index). Each abstract averages 200–300 words (~250–350 tokens), fitting in a single 512-token chunk.

| Tier | Source | Chunks | Qdrant disk | Cumulative |
|------|--------|--------|-------------|------------|
| **T1** | OpenAlex abstracts | ~160M | 672 GB | 672 GB |
| **T2** | PMC OA full text (3.4M articles × 15 chunks) | ~51M | 214 GB | 886 GB |
| **T3** | CORE full text (filtered 10M articles × 15 chunks) | ~150M | 630 GB | 1,516 GB |
| **T4** | CORE full text (all 34M articles × 15 chunks) | ~510M | 2,142 GB | 2,814 GB |

### Complete 3 TB budget allocation (recommended)

| Component | GB | % of 3,072 GB |
|-----------|----|---------------|
| Qdrant vectors + index (T1+T2+T3 = ~361M vectors) | **1,516** | 49% |
| Processed source data (trimmed metadata + text) | **440** | 14% |
| Download cache (temporary, reusable) | **330** | 11% |
| Working buffer + WAL + OS overhead | **200** | 7% |
| **Headroom for incremental growth** | **586** | 19% |
| **Total** | **3,072** | 100% |

**T1+T2+T3 is the sweet spot**: 361M vectors from 173M unique papers (OpenAlex metadata backbone + PMC biomedical full text + 10M selectively filtered CORE full-text papers), consuming ~2.5 TB with 586 GB of growth headroom. Going to T4 (all CORE full text) would produce 670M+ vectors at ~2.8 TB for Qdrant alone, exceeding the budget when source data is included. T4 requires a second drive or aggressive field pruning.

## 2. Download plan: order of operations

**Download OpenAlex first.** It provides the metadata backbone — DOIs, titles, authors, citations, topics, and reconstructable abstracts for 288M works — at zero cost and with no registration. PMC OA comes second because its JATS XML is the highest-quality structured full text available. CORE comes last because it requires registration, is the largest, and overlaps substantially with OpenAlex.

### Phase 1: OpenAlex bulk snapshot

OpenAlex lives in a public S3 bucket with no authentication required. The snapshot contains 11 entity types as gzip-compressed JSON Lines files partitioned by `updated_date`.

```bash
# Check total size before committing (~330 GB)
aws s3 ls --summarize --human-readable --no-sign-request \
  --recursive "s3://openalex/data/works/"

# Download works entity (the only one needed for vector search)
aws s3 sync 's3://openalex/data/works/' './openalex/works/' --no-sign-request

# Also grab topics and sources for enrichment (small)
aws s3 sync 's3://openalex/data/topics/' './openalex/topics/' --no-sign-request
aws s3 sync 's3://openalex/data/sources/' './openalex/sources/' --no-sign-request
```

**Skip these entity types** to save ~100 GB: `authors`, `institutions`, `concepts`, `publishers`, `funders`, `domains`, `fields`, `subfields`. Author and institution names are already embedded in each work record's `authorships` array. Download the `merged_ids` folder for deduplication.

Each partition directory (`updated_date=YYYY-MM-DD/`) contains multiple gzip files under 2 GB each. The manifest file in each entity folder lists all data files in Redshift manifest format.

### Phase 2: PMC Open Access Subset

PMC OA requires no registration. The bulk XML packages provide the highest-quality structured full text for biomedical literature. **Grab the pre-extracted text packages rather than XML if storage is tight** — they're significantly smaller.

```bash
# Download all three license groups (text format, ~40-60 GB total)
# Commercial use (CC0, CC-BY, CC-BY-SA, CC-BY-ND)
wget -r -np -nH --cut-dirs=3 -P ./pmc/oa_comm/ \
  https://ftp.ncbi.nlm.nih.gov/pub/pmc/oa_bulk/oa_comm/txt/

# Non-commercial use (CC-BY-NC, CC-BY-NC-SA, CC-BY-NC-ND)
wget -r -np -nH --cut-dirs=3 -P ./pmc/oa_noncomm/ \
  https://ftp.ncbi.nlm.nih.gov/pub/pmc/oa_bulk/oa_noncomm/txt/

# Other licenses
wget -r -np -nH --cut-dirs=3 -P ./pmc/oa_other/ \
  https://ftp.ncbi.nlm.nih.gov/pub/pmc/oa_bulk/oa_other/txt/

# Get the file list for metadata mapping
curl -o ./pmc/oa_file_list.csv \
  https://ftp.ncbi.nlm.nih.gov/pub/pmc/oa_file_list.csv
```

**Important**: PMC's FTP service is being deprecated in August 2026. Transition to the PMC Cloud Service on AWS for long-term access. Baseline packages are refreshed twice yearly (mid-June, mid-December); daily incremental packages fill the gaps.

If you want structured XML (for section-aware chunking with `<sec>` tags), download the `/xml/` directories instead — they're larger but enable hierarchical chunking with section headers.

### Phase 3: CORE dataset (conditional on remaining budget)

CORE requires free registration at `https://core.ac.uk/documentation/dataset`. Credentials are emailed after registration.

```bash
# After receiving credentials, download the 2024 dataset batches
# CORE 2024 is split into manageable batches (tar.xz format)
# Use aria2 for parallel downloads with resume support
aria2c -x 16 -s 16 --continue=true -i core_download_urls.txt -d ./core/

# Or with wget for simplicity
wget --user=YOUR_USER --password=YOUR_PASS -c -P ./core/ \
  https://core.ac.uk/datadump/2024/batch_001.tar.xz
```

**Filter strategy for CORE within budget**: Don't download everything. The full dataset is ~393 GB compressed (~3.5 TB decompressed), far exceeding the budget. Instead, download only batches containing records with `fullText` fields. Process records in streaming fashion: decompress → check for `fullText` → extract and save if present → discard the rest. Target the **10M highest-value full-text articles** based on having a DOI (for OpenAlex matching), being in English, and being published after 2000.

## 3. Data processing pipeline: from raw dumps to embeddable chunks

The processing pipeline has three stages: **extract** (parse raw formats into normalized records), **join** (match CORE/PMC records to OpenAlex metadata via DOI), and **chunk** (split text into embedding-ready segments).

### OpenAlex processing: reconstructing abstracts from inverted indexes

OpenAlex stores abstracts as `abstract_inverted_index` — a JSON object mapping words to position arrays — for legal reasons. Reconstructing plaintext is straightforward but the field is extremely large in the raw JSON. **Stream-process each gzip file without fully decompressing**.

```rust
// Pseudocode for streaming OpenAlex processing
// Use flate2 for gzip decompression, serde_json for parsing
for gz_file in openalex_works_dir.read_dir() {
    let reader = BufReader::new(GzDecoder::new(File::open(gz_file)?));
    for line in reader.lines() {
        let work: OpenAlexWork = serde_json::from_str(&line)?;
        // Reconstruct abstract from inverted index
        let abstract_text = reconstruct_abstract(&work.abstract_inverted_index);
        // Extract only needed fields
        let trimmed = TrimmedWork {
            id: work.id,
            doi: work.doi,
            title: work.display_name,
            abstract_text,
            year: work.publication_year,
            authors: work.authorships.iter()
                .take(10)  // Cap at 10 authors
                .map(|a| a.author.display_name.clone())
                .collect(),
            source: work.primary_location.map(|l| l.source.display_name),
            cited_by_count: work.cited_by_count,
            topics: work.topics.iter().take(3).map(|t| t.display_name.clone()).collect(),
            oa_status: work.open_access.oa_status,
            language: work.language,
        };
        write_to_processed_db(trimmed)?;
    }
}
```

**Fields to keep**: `id`, `doi`, `title`/`display_name`, reconstructed abstract, `publication_year`, top 10 `authorships` (name only), `primary_location` source name, `cited_by_count`, top 3 `topics`, `open_access.oa_status`, `language`, `type`.

**Fields to discard**: full `authorships` array (keep only names), `counts_by_year`, `biblio` (volume/issue/pages), `referenced_works`, `related_works`, `locations` (keep only `primary_location`), `grants`, `ngrams_url`, all API URLs. Discarding these fields reduces per-record size from ~5–10 KB to ~500 bytes, compressing the total processed dataset to roughly **200 GB** for 288M records.

Use **DuckDB** or **quick-xml** (Rust) for efficient batch processing. DuckDB's `read_ndjson` function can read directly from gzip files on S3 if you want to skip local download.

### PMC XML processing: section-aware text extraction

PMC articles in JATS XML have well-structured `<sec>` elements with `sec-type` attributes (intro, methods, results, discussion). Parse with `quick-xml` or `roxmltree` in Rust.

```rust
// Simplified JATS XML section extraction with roxmltree
let doc = roxmltree::Document::parse(&xml_content)?;
let body = doc.descendants().find(|n| n.has_tag_name("body"));
for section in body.descendants().filter(|n| n.has_tag_name("sec")) {
    let sec_type = section.attribute("sec-type").unwrap_or("unknown");
    let title = section.children()
        .find(|n| n.has_tag_name("title"))
        .map(|n| n.text().unwrap_or(""));
    let paragraphs: Vec<String> = section.children()
        .filter(|n| n.has_tag_name("p"))
        .map(|n| extract_text_recursive(n))
        .collect();
    // Emit section with metadata for chunking
}
```

**Prefer JATS XML over PDF** for PMC articles. The structured XML gives you clean section boundaries, inline citations, and metadata without any layout analysis or OCR. Reserve `pdf_masher` and `xycut-plus-plus` for PDFs that lack XML (primarily CORE documents or supplementary materials).

### DOI matching: joining CORE full text with OpenAlex metadata

The join strategy uses two passes — **exact DOI match first, then fuzzy title matching** for the remainder.

**DOI coverage**: approximately 50% of records in both OpenAlex and CORE have DOIs. Exact DOI matching is deterministic and fast — build a hash set of OpenAlex DOIs, then scan CORE records. For the remaining ~50% without DOIs, use **normalized title matching with year+author blocking**:

1. Normalize titles: lowercase, strip punctuation, collapse whitespace
2. Block by publication year to reduce comparison space from O(n²) to manageable
3. Within each year block, compute `token_sort_ratio` using RapidFuzz (or a Rust equivalent like `strsim`) at threshold ≥ 0.85
4. Verify matches by checking first-author last name agreement

Expected overall match rate: **60–75% of CORE records** will match to OpenAlex, giving you enriched metadata (citations, topics, institution data) for the full-text articles. The unmatched CORE records still get embedded — they just carry CORE-only metadata.

### Chunking strategy: 512 tokens, 50-token overlap, abstracts as first-class chunks

The 2024–2025 benchmark consensus is clear: **recursive character splitting at 512 tokens with 50–100 token overlap** outperforms semantic chunking, sentence-level chunking, and page-level chunking across multiple independent evaluations (FloTorch 2026, NAACL 2025 Vectara study, Chroma Research).

- **Abstracts**: Treat as standalone chunks. Average academic abstract is 200–300 words (~250–350 tokens), fitting cleanly in one 512-token window. Prepend the paper title: `"{title}\n\n{abstract}"`.

- **Body text**: Split at 512 tokens with 50-token overlap. **Prepend section headers** using the Contextual Chunk Headers (CCH) technique: `"{paper_title} > {section_name}\n\n{chunk_text}"`. This significantly improves retrieval accuracy by giving each chunk document-level and section-level context.

- **Chunk count estimates**: An average academic paper (~6,500 tokens of body text) produces ~15 body chunks at 512/50 overlap, plus 1 abstract chunk = **~16 chunks per full-text paper**. Abstract-only papers produce **1 chunk**.

## 4. Qdrant configuration: schema, quantization, and capacity planning

### Collection schema design

A single collection with payload-based filtering is the recommended architecture for single-node deployment. Qdrant handles internal segmentation automatically.

```rust
use qdrant_client::Qdrant;
use qdrant_client::qdrant::*;

let client = Qdrant::from_url("http://localhost:6334").build()?;

client.create_collection(
    CreateCollectionBuilder::new("papers")
        .vectors_config(VectorParamsBuilder::new(768, Distance::Cosine)
            .on_disk(true))                    // originals on mmap
        .quantization_config(ScalarQuantization {
            r#type: ScalarType::Int8 as i32,
            quantile: Some(0.99),
            always_ram: Some(true),            // quantized in RAM
        })
        .hnsw_config(HnswConfigDiffBuilder::default()
            .m(16)
            .ef_construct(200)
            .on_disk(false))                   // HNSW graph in RAM
        .on_disk_payload(true)                 // payload in RocksDB
        .optimizers_config(OptimizersConfigDiffBuilder::default()
            .memmap_threshold(20000)
            .max_segment_size(5_000_000))
).await?;
```

### Payload fields to index

Create payload indexes **before ingestion** so HNSW benefits from filtered-edge optimization:

| Field | Index type | Purpose |
|-------|-----------|---------|
| `doi` | keyword | Exact DOI lookup, deduplication |
| `year` | integer (range + lookup) | Range queries: "papers from 2020–2025" |
| `source` | keyword | Filter by journal/venue |
| `topic` | keyword | Filter by research area |
| `language` | keyword | Filter to English-only results |
| `chunk_type` | keyword | Filter abstract vs body chunks |
| `paper_id` | keyword | Group-by for deduplicating chunks from same paper |

Store `title`, `authors`, and `abstract_preview` (first 200 chars) in payload but **do not index them** — they're too large for keyword indexes and would consume excessive RAM. Use vector search for semantic matching instead.

### Capacity planning for 32 GB vs 64 GB RAM

The critical question: how many vectors can the machine search performantly?

**RAM component breakdown for 361M vectors (T1+T2+T3):**

| Component | Per vector | 361M vectors | In RAM? |
|-----------|-----------|-------------|---------|
| Scalar quantized (int8) | 768 B | **277 GB** | mmap (page cache) |
| HNSW graph (m=16) | 128 B | **46 GB** | ⚠️ Exceeds 64 GB |
| Original vectors (float32) | 3,072 B | 1,108 GB | mmap (disk) |
| Payload indexes | — | ~5 GB | RAM |

**Problem**: At 361M vectors, even just the HNSW graph at m=16 (46 GB) plus overhead exceeds 32 GB and consumes most of 64 GB. Solutions:

**For 64 GB RAM — Option A (recommended): Reduce HNSW m to 12, scalar quant on mmap**
- HNSW in RAM (m=12): 361M × 96 B = **35 GB**
- Payload indexes + overhead: ~5 GB
- Remaining ~24 GB available for OS page cache (caches hot quantized vectors)
- Original vectors served from NVMe via mmap
- Use `rescore=true` with `oversampling=2.0` at query time
- **Search latency**: ~50–200ms on NVMe SSD, acceptable for research use

**For 64 GB RAM — Option B: Product quantization x16, HNSW m=16 in RAM**
- PQ x16 in RAM (always_ram): 361M × 192 B = **69 GB** — too large
- PQ x16 in RAM (m=8): 361M × 192 = 69 GB — still too large
- **PQ x16 works only up to ~250M vectors** in 64 GB

**For 32 GB RAM: Reduce to T1+T2 (~211M vectors), use aggressive settings**
- HNSW in RAM (m=8): 211M × 64 B = **14 GB**
- Payload indexes + overhead: ~5 GB
- Remaining ~13 GB for page cache
- Scalar quantized vectors on mmap (211M × 768 B = 162 GB on disk)
- This works but search latency depends heavily on NVMe IOPS

**For 32 GB RAM with all 361M vectors: put HNSW on disk too**
- HNSW on mmap (m=12): 35 GB on disk, hot portions cached
- Scalar quant on mmap: 277 GB on disk
- Only ~5 GB actively pinned in RAM
- **Search latency**: 200ms–2s depending on cache hit rate. Acceptable for batch research, not interactive search.
- Requires **NVMe SSD** with >100K random read IOPS

### Quantization recommendation

**Use scalar quantization (int8), not product quantization.** Scalar quantization delivers a clean **4× compression** with negligible accuracy loss (99%+ recall maintained) and faster search via SIMD int8 instructions. Product quantization offers 4×–64× compression but introduces meaningful accuracy degradation and slower search (not SIMD-friendly). The extra compression of PQ is only justified if you absolutely cannot fit HNSW in RAM — and for 361M vectors on 64 GB, scalar quantization with a slightly reduced m value is the better tradeoff.

### Bulk ingestion configuration

Disable HNSW during initial bulk upload, then re-enable. This avoids constant graph rebalancing:

```rust
// Step 1: Create collection with m=0 (no HNSW during ingest)
// ... same as above but with .m(0) ...

// Step 2: Bulk upsert all vectors in batches of 1,000-5,000
for batch in chunks.chunks(2000) {
    let points: Vec<PointStruct> = batch.iter().map(|c| {
        PointStruct::new(c.id, c.embedding.clone(), c.payload.clone())
    }).collect();
    client.upsert_points("papers", points).await?;
}

// Step 3: Re-enable HNSW after all data is loaded
client.update_collection(
    UpdateCollectionBuilder::new("papers")
        .hnsw_config(HnswConfigDiffBuilder::default()
            .m(12)
            .ef_construct(200))
).await?;
// Qdrant will rebuild the HNSW index in the background
```

## 5. Embedding strategy: models, runtimes, and parallelization

### Model selection: nomic-embed-text v1.5 wins for this use case

Both nomic-embed-text v1.5 and BGE-base-en-v1.5 produce 768-dimensional vectors, but they differ in ways that matter for academic text:

| Factor | nomic-embed-text v1.5 | BGE-base-en-v1.5 |
|--------|----------------------|-------------------|
| MTEB average | 62.28 | **63.55** |
| Max input tokens | **8,192** | 512 |
| Parameters | 137M | **109M** |
| Matryoshka dims | **64–768** | 768 only |
| License | Apache 2.0 | MIT |

**Choose nomic-embed-text v1.5.** The 1.3-point MTEB gap is negligible, and the **8,192-token context window** is critical. Academic abstracts average 250–350 tokens but some reach 500+, and body chunks at 512 tokens regularly hit BGE's hard limit after prepending section headers (the CCH technique). Nomic handles these without truncation. The Matryoshka support also provides a future escape hatch — if storage becomes critical, you can drop to 512 or 256 dimensions with a single re-export rather than re-embedding everything.

Nomic requires task prefixes: use `search_document: ` when embedding documents for storage, and `search_query: ` when embedding queries at search time.

**Worth monitoring**: `nomic-embed-text-v2-moe` (newer MoE architecture, stronger on BEIR) and `stella_en_400M_v5` (400M params, significantly higher quality at ~72 MTEB but 3× the compute cost). If embedding throughput proves faster than expected, stella is the upgrade path.

### Runtime selection: ONNX Runtime via the `ort` crate

| Runtime | Throughput | GPU support | Rust integration | Verdict |
|---------|-----------|-------------|-----------------|---------|
| **ONNX Runtime (`ort`)** | **Highest** | CUDA, TensorRT, CoreML, Metal | Native via `ort` crate | **Best for batch embedding** |
| TEI (HuggingFace) | High | CUDA, Metal | HTTP API (not embedded) | Best as a server, overkill for batch |
| Candle | Moderate | CUDA, Metal | Native Rust | 3–4× slower GPU than ONNX |
| fastembed-rs | High | Via `ort` backend | Native Rust | Simpler API, good alternative |
| Ollama | Low | CUDA, Metal | HTTP API | Known batch quality bugs — avoid |

**Use ONNX Runtime** via the `ort` crate for maximum throughput. Convert nomic-embed-text v1.5 to ONNX format (or use pre-converted models from HuggingFace), enable FP16 inference and O3 optimization level. The `ort` crate provides direct CUDA execution provider support for the RTX 3090 and CoreML/Metal execution providers for MacBooks.

**Avoid Ollama** for batch embedding — GitHub issue #6262 documents embedding quality degradation at batch sizes ≥16.

### Batch sizing for 24 GB VRAM (RTX 3090)

For nomic-embed-text v1.5 (137M params) with FP16 inference:
- Model weights: ~260 MB
- Per-batch activation memory at 512 tokens: ~(batch × 512 × 768 × 12 layers × 2 bytes × 3) ≈ batch × 27 MB
- **Optimal batch size at 512 tokens: 64–128** (uses ~2–4 GB activations + 260 MB model = ~2.5–4.5 GB, well within 24 GB)
- At 256 tokens: batch size 128–256
- At 128 tokens: batch size 256–512
- **Always use FP16** — it leverages the RTX 3090's Tensor Cores for a **1.5–2× speedup** with no meaningful quality loss

### Throughput estimates and time to completion

**Conservative throughput estimates** (validated against community benchmarks for BERT-base-class models):

| Device | Seq length 256 | Seq length 512 | Notes |
|--------|---------------|----------------|-------|
| **RTX 3090** (FP16, ONNX) | ~2,500 emb/s | ~1,500 emb/s | Tensor Cores active |
| **M2/M3 Max MacBook** (Metal) | ~800 emb/s | ~500 emb/s | ~30–40% of 3090 |
| **M1/M2 Pro MacBook** (Metal) | ~500 emb/s | ~300 emb/s | ~20–25% of 3090 |

**Combined fleet throughput** (RTX 3090 + 3 MacBooks averaging 600 emb/s each):
- At 512-token chunks: 1,500 + 1,800 = **~3,300 embeddings/sec**
- With **30% I/O overhead**: ~2,300 effective embeddings/sec

| Vector count | RTX 3090 only | Full fleet (3090 + 3 MacBooks) |
|-------------|--------------|-------------------------------|
| 160M (T1) | ~30 hours | ~19 hours |
| 211M (T1+T2) | ~39 hours | ~25 hours |
| 361M (T1+T2+T3) | ~67 hours | ~44 hours |

### Parallelization architecture: file-based partitioning

The simplest approach that works: **pre-partition the processed chunk files by machine capacity**.

```
processed_chunks/
├── rtx3090/          # 55% of files (highest throughput)
│   ├── batch_0001.jsonl
│   ├── batch_0002.jsonl
│   └── ...
├── macbook_1/        # 15% of files
├── macbook_2/        # 15% of files
└── macbook_3/        # 15% of files
```

Each machine runs the same embedding binary (compiled for its target: CUDA for RTX 3090, Metal/CoreML for MacBooks). Outputs are written as `(point_id, vector)` pairs to local disk, then `rsync`'d to the Qdrant host for batch insertion. Over gigabit LAN, transferring 361M embeddings (361M × 3 KB = ~1 TB) takes ~2.5 hours — negligible compared to embedding time.

For more dynamic load balancing, use a Redis or NATS queue where each machine pulls the next batch of chunks to embed. This handles heterogeneous speeds gracefully but adds operational complexity.

**Optimization tips that matter**: sort chunks by token count before batching (minimizes padding waste, can improve throughput 20–30%), pre-tokenize on CPU while GPU processes the previous batch (overlap I/O and compute), and pin the embedding model in GPU memory across batches (avoid reloading).

## 6. Incremental update strategy: keeping the index fresh

Each data source has a different update cadence. The key principle: **track the last-synced timestamp for each source and only process records newer than that timestamp**.

### OpenAlex: updated_date partitions (daily/weekly)

OpenAlex's partition-by-`updated_date` design makes incremental sync trivial. When a record changes, it moves from its old partition to a new one — you never need to re-download old partitions.

```bash
# Sync only partitions newer than your last sync date
aws s3 sync 's3://openalex/data/works/' './openalex/works/' \
  --no-sign-request \
  --exclude "*" \
  --include "updated_date=2026-03-*/*"
```

Process new/updated records through the same pipeline: reconstruct abstract → chunk → embed → upsert into Qdrant (upsert overwrites existing points with the same ID, handling updates automatically). Also process the `merged_ids` folder: delete any merged entity IDs from Qdrant. OpenAlex adds roughly **50,000 new works per day**, meaning ~350K new abstract chunks per week — embeddable in under 5 minutes on the RTX 3090.

### PMC OA: baseline + daily incrementals

PMC publishes daily incremental packages (`oa_comm_txt.incr.YYYY-MM-DD.tar.gz`) between biannual baselines. Download and process incrementals daily or weekly:

```bash
# Download today's incremental
wget https://ftp.ncbi.nlm.nih.gov/pub/pmc/oa_bulk/oa_comm/txt/oa_comm_txt.incr.$(date +%Y-%m-%d).tar.gz
```

When a new baseline drops (mid-June, mid-December), all previous baselines and incrementals are deleted. Re-download the baseline and diff against your existing processed data.

### CORE: annual snapshots + API supplementation

CORE releases public datasets approximately annually. Between releases, use the CORE API (free tier: 1 batch or 5 single requests per 10 seconds) to check for new full-text articles matching high-priority criteria. The FastSync service provides real-time updates but requires membership.

### Re-embedding strategy

When documents change, **re-embed only affected chunks**. Use a content hash (e.g., xxHash of the chunk text) stored in Qdrant payload. On update: recompute the hash → if unchanged, skip embedding → if changed, re-embed and upsert. For OpenAlex abstract updates (the most common change), this avoids re-embedding unchanged records that merely had metadata updates (citation count changes, topic reassignment).

## 7. What to cut when 3 TB runs tight

Priority tiers in order of **what gets sacrificed first**:

1. **Cut first — CORE body text beyond 10M articles (T4→T3)**: Savings of ~1,300 GB in Qdrant. CORE's full 34M articles produce 510M body chunks — the single largest storage consumer. Filtering to 10M high-value articles (English, post-2000, has DOI, high citation count) retains 80%+ of the retrieval value at 30% of the cost.

2. **Cut second — Original vectors in Qdrant**: If you disable rescoring (`rescore=false`), Qdrant still stores originals on disk but you could theoretically not allocate SSD space for them. In practice, Qdrant always stores originals, but you can move them to slower storage (HDD) since they're only accessed during rescoring. This "saves" ~1.1 TB of NVMe space for 361M vectors at the cost of no rescoring capability.

3. **Cut third — Reduce HNSW m from 12 to 8**: Saves 361M × 32 B = ~12 GB of RAM and proportional disk. Small recall impact (~1–2% at typical ef values).

4. **Cut fourth — Drop Matryoshka dimensions from 768 to 512**: If using nomic-embed-text v1.5, reduce output dimensions from 768 to 512. Per-vector disk drops from 4.2 KB to ~3.0 KB — a **29% reduction** across all vectors. At 361M vectors, this saves ~430 GB. Requires re-embedding but nomic's Matryoshka training ensures minimal quality loss at 512 dims.

5. **Cut fifth — Drop PMC OA (T2)**: Saves ~214 GB in Qdrant + 40 GB source data. PMC's 3.4M articles are high-quality but the content overlaps significantly with CORE. Cut PMC if you're keeping a large CORE slice.

6. **Never cut — OpenAlex abstracts (T1)**: The metadata backbone. 160M abstract vectors at 672 GB is the foundation. Every other source enriches this core.

If your actual available space is only 2 TB (after OS, swap, and system files on a nominal 3 TB drive):

## Existing projects and 2024–2025 best practices worth studying

Several production systems validate this architectural approach. **Semantic Scholar** (Allen Institute for AI) indexes 225M+ papers using SPECTER2 embeddings with a hybrid dense + sparse (BM25) retrieval pipeline, storing 275M passages from 12M full-text papers at 480 tokens per passage with 1-sentence overlap. Their production system uses FAISS for approximate k-NN with binary quantization on `mxbai-embed-large-v1` embeddings.

The **LAION-400M benchmark** from Qdrant's own team is the closest architectural reference: 400M vectors (512-dim CLIP) served on a single 64 GB RAM machine using `on_disk=true`, binary quantization with `always_ram=true`, and `m=6`. This proves Qdrant can handle the scale.

For the Rust ecosystem specifically, the key crates are `qdrant-client` (v1.12.1, gRPC-based, production-ready), `ort` (ONNX Runtime bindings for embedding inference), `quick-xml` or `roxmltree` (JATS XML parsing — no dedicated Rust JATS parser exists), `tokenizers` (HuggingFace's fast tokenizer), and `flate2` (gzip decompression for streaming OpenAlex processing). The `fastembed-rs` crate provides a higher-level embedding API built on `ort` that supports BGE-base-en-v1.5 out of the box and nomic-embed-text-v2-moe.

Open-source RAG frameworks to study include **RAGFlow** (deep document understanding), **Kotaemon** (hybrid retriever with reranking), and **txtai** (sparse + dense + graph embedding database). None of these operate at the 100M+ vector scale described here, which makes the home-still project genuinely novel in the open-source space.

## Conclusion

The most important architectural decision is **not which data to download, but how aggressively to chunk it**. OpenAlex's 160M abstracts at one chunk each are cheap; CORE's 34M full-text articles at 15 chunks each are the budget-breaker. The recommended T1+T2+T3 configuration (361M vectors from OpenAlex abstracts + PMC full text + 10M filtered CORE articles) consumes ~2.5 TB and provides semantic search across 173M unique papers — the largest open-source academic search index documented to date.

Three decisions lock in early and are expensive to reverse: **embedding model** (nomic-embed-text v1.5 for its 8K context window), **vector dimensionality** (768, with Matryoshka fallback to 512), and **Qdrant quantization strategy** (scalar int8 with mmap originals). Get these right and everything else — download ordering, chunking parameters, HNSW tuning — can be iterated incrementally as the index grows.

The processing timeline is dominated by embedding generation: roughly **44 hours wall-clock** for the full 361M vectors across the RTX 3090 and three MacBooks. Downloads take 12–24 hours depending on bandwidth. The entire prototype — from first `aws s3 sync` to first vector search query — is achievable in **under one week**.
