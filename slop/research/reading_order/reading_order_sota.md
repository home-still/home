# State-of-the-art reading order algorithms for PDF document extraction

**Reading order detection has achieved 98.8% BLEU accuracy on complex multi-column layouts as of 2024**, with transformer-based models like LayoutLMv3 and LayoutReader establishing the foundation while recent hybrid approaches like XY-Cut++ deliver production-ready performance. The field has converged on three complementary approaches: transformer-based sequence-to-sequence models for maximum accuracy, graph neural networks for 10-100× faster inference with minimal accuracy loss, and hybrid heuristic-ML systems balancing both. MinerU specifically integrates LayoutReader for reading order detection in its pipeline backend while achieving state-of-the-art results through its vision-language model backend that learns reading order natively during training.

The breakthrough came from Microsoft Research's ReadingBank dataset (500,000 documents with automated reading order annotations) enabling LayoutReader to achieve near-perfect reading order detection. However, practical deployment faces a critical accuracy-speed tradeoff: sequence-to-sequence models require 687ms per page while graph-based approaches process 244 pages per second at competitive accuracy. For production systems, ONNX deployment with INT8 quantization provides 2-4× speedup with <1% accuracy loss, and Rust implementations eliminate Python runtime overhead entirely.

## Transformer-based models dominate current benchmarks

The foundation of modern reading order detection rests on three architectures that revolutionized document understanding between 2021-2024. **LayoutReader** introduced the first large-scale deep learning approach specifically designed for reading order, using a sequence-to-sequence architecture with a LayoutLM-based encoder that captures both text and spatial layout through 2D bounding box coordinates. The model processes text tokens combined with normalized position embeddings (x, y, width, height in 0-1000 range) and uses autoregressive decoding to predict the correct sequential order of arbitrarily shuffled text segments. Training on ReadingBank's 500,000 images enabled near-perfect performance, reducing Average Relative Distance by 6.71 compared to baselines and improving BLEU scores by 0.2847. Critically, ablation studies revealed that layout information contributes more to reading order accuracy than text content itself—layout-only models achieved 0.27 BLEU improvement versus 0.16 for text-only.

**LayoutLMv3** (ACM MM 2022) represents the architectural evolution that eliminated CNN backbones entirely. The BASE model contains 12 layers, 768 hidden dimensions, and 133 million parameters—44 million fewer than LayoutLMv2 by using direct ViT-style linear projection of 16×16 image patches rather than ResNet-101 feature extraction. This multimodal transformer processes three input types simultaneously: RoBERTa-initialized text embeddings with 2D segment-level bounding boxes, linearly-projected image patches from 224×224 resolution documents, and unified layout embeddings shared across words in the same semantic segment. The pre-training regime combines masked language modeling (30% span masking with Poisson λ=3 distribution), masked image modeling (40% blockwise patch masking reconstructing discrete 8,192-vocabulary visual tokens), and word-patch alignment (novel cross-modal objective predicting whether image patches are masked). Trained on 11 million pages from IIT-CDIP over 500,000 steps with batch size 2,048, LayoutLMv3-BASE achieves 90.29% F1 on FUNSD form understanding and 95.1% mAP on PubLayNet layout analysis. While not explicitly designed for reading order, its segment-level layout positions and cross-modal alignment provide strong foundations—LayoutReader can be built on LayoutLMv3 backbones for significantly faster inference than the original seq2seq architecture.

**DiT (Document Image Transformer)** from CVPR 2022 takes a vision-only approach with pure ViT architecture: 12-layer BASE (86M parameters) and 24-layer LARGE (304M parameters) models process 224×224 images divided into 196 patches of size 16×16. Pre-trained via masked image modeling using a discrete VAE tokenizer with 8,192 visual tokens on 42 million document pages, DiT learns document layout patterns through self-supervision without any text modality. The model achieves 92.69% accuracy on RVL-CDIP document classification and 94.9% mAP on PubLayNet layout detection. Its discrete visual tokenizer became integral to LayoutLMv3's masked image modeling objective, and DiT serves as the visual backbone in multimodal systems requiring strong layout understanding for reading order prediction.

Recent advances in 2023-2025 pushed beyond these foundations. **DLAFormer** (CVPR 2024) introduced unified end-to-end transformers treating reading order as a relation prediction problem rather than sequence generation, enabling concurrent processing of table detection, text region identification, logical role classification, and reading order prediction through type-wise queries and coarse-to-fine strategies. The hierarchical document structure analysis framework follows a detect-order-construct pipeline: transformer-based object detection (DINO/DETR variants) identifies page elements, a reading order module predicts sequential relationships between detected objects, and tree construction algorithms build hierarchical structures with parent-child relationships. Benchmarks on HRDoc (66K documents) and Comp-HRDoc (42K documents with 14 hierarchical categories and 3 relation types) demonstrate state-of-the-art performance handling complex academic papers with nested sections, subsections, and cross-references.

Vision-language models emerged as a fourth approach in 2024-2025, implicitly learning reading order through visual instruction tuning rather than explicit sequence prediction. Models like Idefics3-8B trained on the Docmatix dataset (240× larger than previous document VQA datasets), Qwen-VL with text localization capabilities, and mPLUG-DocOwl enabling OCR-free understanding all demonstrate reading order inference from pure visual layout analysis. These systems combine CLIP-based or SigLIP vision encoders (increasingly using efficient FastViT achieving 8× smaller size and 20× faster speed than ViT-L/14), projection layers or Q-Formers for visual token abstraction, and large language models (LLaMA, Qwen, Mistral variants with 8K-128K context windows) instruction-tuned for visual tasks. While less interpretable than explicit sequence models, VLMs handle multi-column layouts through holistic visual understanding and can be prompted to describe reading flow, making them particularly effective for diverse document types where rule-based heuristics fail.

## Graph neural networks offer compelling efficiency advantages

Graph-based approaches model documents as networks where text blocks become nodes and spatial relationships become edges, offering 10-100× faster inference than transformers with only 5-10% accuracy reduction. **PARAGRAPH2GRAPH** (2023) uses Dynamic Graph Convolutional Neural Networks with just 19.95 million parameters—6.7× smaller than LayoutLMv3's 133M—achieving competitive results on DocLayNet and PubLayNet without requiring tokenizers or language-specific preprocessing. Nodes receive features from pre-trained ResNet-18 ROI pooling (visual features), normalized 2D OCR coordinates (geometric features), text content and metadata (font size, character counts, aspect ratios), while edges connect k-nearest neighbors in each cardinal direction (typically k=1) with direction and distance features. This language-independent design handles arbitrarily long documents without transformer sequence length constraints.

**GLAM (Graph-based Layout Analysis Model)** demonstrates the speed advantage: with only 4 million parameters (1M for the GNN component alone), GLAM processes pages in 10ms on GPU—68.7× faster than LayoutLMv3's 687ms and 5.6× faster than YOLOv5x6's 56ms—while achieving 68.6% mAP on DocLayNet test set. The two-stage approach first performs node classification for semantic labels using Graph Convolutional Networks with Topology Adaptive Graph convolutional layers, then conducts edge classification to determine segment boundaries. GLAM excels at text-based elements (titles, headers) with pixel-perfect boundary alignment but struggles with image-rich regions (figures, pictures scoring 56.0% and 59.1% AP respectively versus 88.1% for text). When ensembled with YOLOv5x6, the combined system achieves 80.8% mAP—a new state-of-the-art exceeding either model alone, demonstrating that GNN spatial reasoning complements transformer visual understanding.

**ROPE (Reading Order Equivariant Positional Encoding)** from ACL 2021 addresses a fundamental GCN limitation: conventional graph neural networks struggle to capture sequential presentation order. ROPE generates unique reading order codes for neighboring words relative to the target word, with equivariant properties ensuring codes remain unchanged if all neighbors and target shift equally in the document. This encoding improves existing GCNs by up to 8.4% F1-score and shows robustness to OCR reading order errors that plague sequence-based models. Combined with pointer network decoders that enable parallel decoding rather than autoregressive generation, graph-based reading order systems achieve the speed necessary for production deployment while maintaining interpretability—the explicit graph structure allows debugging of spatial relationship errors impossible with black-box transformer attention.

The graph construction process critically impacts performance. Nodes typically represent text boxes extracted via PDF parsers (pdfminer, PDFPlumber) or OCR engines (Tesseract, Azure Document Intelligence), with features including bounding box coordinates, text content length, font information, and CNN-derived visual features. Edges connect blocks using k-nearest neighbors (4-8 edges per node), distance thresholds, visibility graphs (vertically/horizontally overlapping blocks), or heuristic reading order connections (left-to-right, top-to-bottom). The resulting directed or undirected graphs remain sparse, enabling efficient message passing through Graph Attention Networks or GraphSAGE architectures that aggregate information from spatial neighbors to classify reading order relationships.

Recent innovations combine graph benefits with deep learning power. **GraphLayoutLM** (2023) extends LayoutLMv3 with Graph Attention Networks establishing edges based on spatial proximity, semantic similarity, and reading order flow, applying graph reordering before the multimodal transformer. **GraphDoc** (2024) introduces comprehensive graph-based structure analysis on 80,000 images with 4.13 million relation annotations across 8 relation categories, adapting Scene Graph Generation techniques for documents to model spatial and logical relations jointly. The **MLARP (Multi-modal Layout-Aware Relation Prediction)** framework addresses semantic adjacency versus spatial proximity by performing binary relationship prediction between text blocks with adaptive distance measures for semantically related but spatially distant text.

## MinerU implements production-ready reading order detection

MinerU, developed by Shanghai AI Laboratory's OpenDataLab, exemplifies modern document extraction architecture through its dual-backend design. The **pipeline backend** uses specialized atomic models for each task—DocLayout-YOLO for layout detection, PaddleOCR for text recognition, YOLOv8 for formula detection, RapidTable/StructEqTable for tables—with **LayoutReader explicitly handling reading order sorting** as a post-processing step. The four-stage workflow processes PDFs through preprocessing (metadata extraction, language identification), content parsing (layout analysis, OCR), post-processing (bounding box resolution, **reading order sorting**, paragraph stitching), and format conversion to Markdown/JSON. Version 0.9.0 (October 2024) marked the critical transition, with release notes stating: "Refactored the sorting module code to use layoutreader for reading order sorting, ensuring high accuracy in various layouts."

The implementation uses the third-party `ppaanngggg/layoutreader` package—a faster LayoutLMv3-based variant rather than the original Microsoft seq2seq architecture. This LayoutLMv3ForTokenClassification approach performs single-pass prediction on bounding boxes in 0-1000 normalized coordinate range, outputting ordered sequence indices representing reading order. The integration occurs after layout detection produces bounding boxes with type labels and content recognition fills in text/formulas/tables: post-processing resolves overlaps (removing contained elements like formulas within tables, shrinking partially overlapping boxes), LayoutReader sorts the cleaned bounding boxes, then paragraph stitching applies cross-column and cross-page logic. The output `content_list.json` stores blocks sequentially with type, text, text_level (for heading hierarchy), bbox, and page_idx fields—this ordered list directly represents reading order for Markdown generation.

The **VLM backend** introduced in MinerU 2.0 (June 2025) takes a radically different approach: **reading order sorting emerges natively from the 1.2 billion parameter multimodal model** rather than requiring a separate module. The technical specification describes "Multiple Functions in One: A single model covers multilingual recognition, handwriting recognition, layout analysis, table parsing, formula recognition, reading order sorting, and other core tasks." This end-to-end vision-language transformer with two-stage inference (layout analysis decoupled from content recognition) and native high-resolution processing achieves state-of-the-art performance on OmniDocBench benchmark, surpassing Gemini 2.5 Pro, GPT-4o, and Qwen2.5-VL-72B. The MinerU 2.5 model (September 2025) outputs content blocks already sorted in reading order without explicit sorting steps, demonstrating superior accuracy on complex layouts compared to the pipeline backend's LayoutReader approach.

Performance benchmarks reveal strengths and limitations. MinerU ranks second among open-source tools on OmniDocBench (behind commercial Mathpix), excelling on English documents and academic papers. Speed comparisons show **0.21 seconds per page with GPU acceleration** (fastest among competitors) versus 3.3 seconds on CPU-only. However, known limitations persist: the technical documentation acknowledges "Reading order is determined by the model based on the spatial distribution of readable content, and may be out of order in some areas under extremely complex layouts," particularly struggling with 3+ column layouts, vertical text (limited support improving in v2.1.0+), and special document types like comic books, art albums, and handwritten notes where layout assumptions break down. GitHub issues (#3591) report wrong reading order in two-column PDFs, and OmniDocBench evaluation shows accuracy decreases as layout complexity increases—single-column performs well, double-column shows slight degradation, three+ columns see notable accuracy drops.

The architectural choice between pipeline and VLM backends reflects fundamental tradeoffs. The pipeline backend offers transparency (LayoutReader's explicit sorting allows debugging), CPU compatibility, modular debugging, and granular control over processing steps, making it suitable for simple to moderate layouts and environments requiring interpretability. The VLM backend delivers higher accuracy through emergent reading order behavior learned during training, faster processing with GPU acceleration (though requires 8GB+ VRAM and Turing architecture GPUs), and superior handling of complex or diverse documents, but operates as a monolithic black box less amenable to fine-grained debugging. The roadmap indicates ongoing development of dedicated reading order models in the PDF-Extract-Kit project, suggesting recognition that further improvements require specialized architectures beyond both current approaches.

## Hybrid and rule-based methods achieve practical performance

Classical algorithms remain relevant through continuous refinement. The **XY-Cut algorithm** recursively partitions documents using horizontal and vertical projection profiles, dividing pages into smaller regions based on white space patterns. While fast and effective for clean layouts, XY-Cut fails catastrophically on "L-shaped" structures, cross-layout configurations, and documents without clear connectivity of blocks—errors propagate through recursive steps, yielding 61.7% Position Error Rate on complex documents. The **Optimized XY-Cut** (2005) improved computational tractability using bounding boxes of connected components rather than pixel-level processing, achieving 89% recall on heterogeneous documents but still fundamentally limited by recursive partitioning assumptions.

**XY-Cut++** (2024) represents the state-of-the-art hybrid approach, achieving **98.8% BLEU score on DocBench-100 benchmark**—a 24% improvement over baselines and matching or exceeding deep learning methods on specific complex layouts. The four-stage pipeline combines heuristics with shallow semantic information: PP-DocLayout extracts visual features and semantic labels, pre-masking removes dynamic elements (titles, tables, figures) before segmentation, multi-granularity segmentation applies adaptive density-based splitting accounting for variable text density within columns, and cross-modal matching uses IoU-weighted margins for caption-to-figure alignment. This lightweight interpretable system addresses traditional XY-Cut failures (multi-column newspapers, mixed text/table/figure arrangements, academic papers with complex sectioning) while maintaining computational efficiency suitable for production deployment.

The **Abstract Argumentation approach** (2014) uses unsupervised techniques based on argumentation theory and general assumptions about human reading behavior, requiring less background knowledge than supervised learning but limited to documents following standard conventions. **Parsr** from AXA Group implements production-ready XY-cut with vertical cut detection between elements, horizontal cut processing for left/right parts, and re-assembly logic preventing text column splitting. This REST API-based modular framework handles left-to-right languages effectively but requires adaptation for right-to-left scripts and may fail with large spacing between bounding boxes.

Recent hybrid innovations integrate machine learning selectively. **Group Position Embedding** (ICLR 2025) encodes document layout coordinates directly into LLMs without additional computational overhead, achieving layout understanding without pre-training through efficient positional encoding. The **MLARP framework** balances heuristic spatial relationships with learned semantic adjacency, addressing cases where semantically related text appears spatially distant due to page layout constraints. **HDT (Hierarchical Document Transformer)** introduces auxiliary anchor tokens enabling multi-level sparse attention (sentence → paragraph → section hierarchy) that reduces quadratic transformer complexity while exploiting document structure—sample-dependent hierarchical attention patterns adapt to each document's specific organization.

## Training datasets enable and constrain algorithm development

**ReadingBank** provides the foundation for supervised reading order detection: 500,000 document images automatically extracted from Microsoft WORD XML metadata with embedded reading order in the document structure. The automated generation from DocX files ensures high-quality annotations at scale impossible with manual labeling, covering diverse document types (receipts, forms, multi-column layouts, invoices) with 400K/50K/50K train/validation/test splits. This dataset enabled LayoutReader's breakthrough performance, though the WORD template origins limit layout variability compared to PDFs from diverse sources. The average BLEU score distribution across the dataset reveals difficulty ranges, helping researchers identify challenging document types requiring additional focus.

**DocLayNet** represents the gold standard for layout diversity: 80,863 manually annotated pages from financial reports, manuals, scientific articles, laws, patents, and government tenders. Forty annotators over three months labeled 11 categories (Caption, Footnote, Formula, List-item, Page-footer, Page-header, Picture, Section-header, Table, Text, Title) with multiple annotations enabling inter-annotator agreement measurement. The human baseline of **82-83% mAP** establishes the performance ceiling—current best models achieve 76.8% mAP (YOLOv5x6), leaving a ~10% gap attributable to genuine ambiguity or annotation inconsistency. Document-wise train/test splits prevent overfitting to specific templates, and learning curves show mAP increases linearly with log(data size) plateauing around 80% of the full dataset—data diversity matters more than absolute size. Cross-dataset evaluation reveals DocLayNet-trained models generalize significantly better than PubLayNet or DocBank models, validating the investment in layout variability.

**PubLayNet** and **DocBank** serve specialized scientific document analysis: PubLayNet's 360,000 images from PubMed Central articles cover 5 categories (Text, Title, List, Table, Figure) with automated XML-to-PDF matching achieving 93%+ mAP but poor transfer to diverse layouts. DocBank's 500,000 pages from arXiv LaTeX documents provide token-level annotations for 13 categories (abstract, author, caption, equation, figure, footer, list, paragraph, section, table, title, reference, date), enabling fine-grained information extraction but limited to academic paper templates. Both datasets demonstrate the power of automated annotation from structured source formats (XML, LaTeX) but reveal the brittle generalization of models trained exclusively on homogeneous layouts.

**HRDoc and Comp-HRDoc** (2023-2024) extend beyond flat layout to hierarchical structure: 66K and 42K documents respectively with 14 categories including sections, subsections, and paragraphs, plus 3 relation types (reading order, parent-child, reference). Line-level annotations in PDF format enable training systems that construct tree structures representing document organization—reading order becomes one dimension of multi-relational understanding including containment hierarchies and cross-reference graphs. The **GraphDoc dataset** (2024) pushes this further with 80,000 images containing 4.13 million relation annotations across 8 relation types, enabling comprehensive graph-based structure analysis.

Synthetic data generation addresses annotation costs while introducing risks. **DocSynth300K** used for DocLayout-YOLO pre-training creates 300,000 synthetic documents with programmatically defined layouts, enabling models to achieve 81.7% AP50 on D4LA and 93.0% AP50 on DocLayNet with transfer learning. However, synthetic documents lack the layout quirks, degradations, and inconsistencies of real-world PDFs—models trained exclusively on synthetic data often fail on scanned documents, watermarked pages, or creative layouts deviating from template assumptions. The optimal strategy combines large-scale synthetic pre-training with fine-tuning on smaller real-world annotated datasets, balancing data efficiency with robustness.

## Evaluation metrics reveal different aspects of reading order quality

**BLEU (Bilingual Evaluation Understudy)** scores measure n-gram overlap between predicted and ground truth reading order sequences, ranging 0-1 with higher values indicating better agreement. Page-level BLEU averages across all pages, providing the primary metric for reading order accuracy. XY-Cut++ achieving 98.8% BLEU represents near-perfect ordering, while LayoutReader's 0.2847 BLEU improvement over baselines demonstrates significant gains. However, BLEU treats all errors equally—placing an element one position wrong versus ten positions wrong receives similar penalty, motivating complementary metrics.

**Average Relative Distance (ARD)** quantifies how far elements deviate from correct positions, measuring not just correctness but severity of errors and estimating manual correction effort. LayoutReader reduces ARD by 6.71 compared to baselines, meaning predicted orders average nearly 7 positions closer to ground truth. Good systems achieve ARD ~6.23 per document on ReadingBank. ARD proves particularly valuable for production systems where understanding error severity helps prioritize fixes—a transposed footnote matters less than a misplaced paragraph breaking narrative flow.

**Position Error Rate (PER)** measures the percentage of incorrectly positioned blocks in reading order, offering intuitive interpretation. Recent learning-to-rank approaches achieve 5.4% PER versus 61.7% for heuristic methods and 56.5% for MLP baselines, demonstrating the superiority of sophisticated ranking models over naive classification or handcrafted rules. PER's block-level granularity makes it suitable for layout-centric evaluation where individual element placement matters more than sequence coherence.

**Spearman's Footrule Distance and Kendall's Tau** provide ranking correlation metrics from statistics, measuring concordance between predicted and ground truth orderings through pairwise comparisons. Kendall's Tau proves particularly robust to outliers, considering whether pairs of elements appear in the correct relative order rather than absolute positions. These metrics shine in handwritten document evaluation where reading order may have multiple valid interpretations—a high Kendall's Tau with moderate BLEU suggests the predicted order respects most pairwise relationships despite differing from the specific ground truth sequence.

**mean Average Precision (mAP)** remains standard for layout detection evaluation underlying reading order systems. mAP@0.5-0.95 (COCO-style) averages precision across IoU thresholds from 0.5 to 0.95, while mAP@0.5 uses a single threshold. Human inter-annotator agreement on DocLayNet reaches 82-83% mAP, while best models achieve 72-77% mAP—the persistent 10% gap suggests either genuine ambiguity in layout interpretation or limitations in current architectures. Per-class performance varies dramatically: Text (88.1%), Table (86.3%), and List-item (86.2%) classify easily due to visual distinctiveness, while Page-footer (61.1%) and Formula (66.2%) struggle due to subtle visual differences and context dependence.

**F1 scores** dominate token classification and information extraction tasks. LayoutLM achieves 79.27% F1 on FUNSD form understanding and 95.24% F1 on SROIE receipt parsing, demonstrating high accuracy on structured documents with clear field boundaries. However, traditional F1 proves sensitive to reading order errors—extracting fields in wrong sequence tanks performance even if individual extractions are correct. **Reading Order Independent Metrics** address this by matching entities independent of sequence, better reflecting downstream task performance when order doesn't strictly matter for the application.

ICDAR competitions provide standardized benchmarks: ICDAR 2024's "Reading Documents Through Aria Glasses" Task B challenged reading order prediction from low-resolution AR glass inputs (winner achieved 0.0939 BLEU on difficult data), while ICDAR 2024's handwritten document task evaluated page-level recognition with PCRR (Page-level Character Recognition Rate) and PWRR (Page-level Word Recognition Rate). The winner achieved 77.44% PCRR and 50.55% PWRR, showing substantial difficulty in preserving reading order during handwriting recognition—character-level accuracy significantly exceeds word-level accuracy, suggesting boundary detection and word formation remain challenging.

## ONNX deployment and Rust integration enable production systems

ONNX (Open Neural Network Exchange) bridges research models and production inference through standardized model representation and runtime optimization. **YOLO-based document models** offer the smoothest ONNX path: models like `Oblix/yolov10m-doclaynet_ONNX_document-layout-analysis` and `Oblix/yolov8x-doclaynet_ONNX` ship pre-exported in ONNX format, working directly with ONNX Runtime and Transformers.js for JavaScript/TypeScript deployment. These models detect Text, Title, List-item, Section-header, Page-header, Page-footer, Table, Picture, and Formula with production-ready performance.

**LayoutLM** (v1) enjoys full official ONNX export support through Hugging Face Transformers and Optimum. The export process requires installing `transformers[onnx]` and running `python -m transformers.onnx --model=microsoft/layoutlm-base-uncased onnx/` to generate ONNX model files. LayoutLMv2 and LayoutLMv3 face partial support challenges—GitHub issue #14368 documents custom configuration requirements, and community workarounds exist but lack official support. Intel OpenVINO provides an alternative conversion path through `openvino-dev` tools converting ONNX models to Intermediate Representation (IR) format optimized for Intel hardware.

**Quantization** delivers critical production optimization: dynamic quantization converts weights to INT8 with runtime activation quantization requiring no calibration data, while static quantization quantizes both weights and activations using calibration data for maximum performance. Hugging Face Optimum simplifies this through `ORTQuantizer` with configuration options like `AutoQuantizationConfig.avx512_vnni()` targeting specific CPU instruction sets. Command-line usage `optimum-cli onnxruntime quantize --onnx_model model_dir --avx512_vnni --output quantized_model` produces optimized models achieving **2-4× smaller size, 2-3× faster inference on CPU with proper instructions, ~2.2× memory reduction, and <1% accuracy loss** with proper calibration. The speed gains prove essential for CPU-only deployment scenarios where GPU acceleration isn't available.

**Rust integration** eliminates Python runtime overhead for edge deployment and embedded systems. The `layoutparser-ort` crate (https://lib.rs/crates/layoutparser-ort) provides a production-ready Rust port running Detectron2 and YOLOX layout models in ONNX format through `ort` crate bindings to onnxruntime. With just 4 million parameters and native Rust performance, this approach suits resource-constrained environments requiring deterministic latency and minimal memory footprint. PDF processing in Rust relies on mature libraries: **lopdf** (pure Rust implementation for reading, modifying, writing PDFs with text extraction, bounding box extraction, object stream support, and automatic encrypted PDF decryption requiring Rust 1.85+), **pdf-rs/pdf** (read, alter, write capabilities with experimental writing and hierarchical primitive visualization), and **printpdf** (create, read, write, render PDFs with pure Rust implementation and MIT license).

A typical Rust document AI pipeline loads PDFs via `lopdf::Document`, extracts page images, initializes ONNX layout models through `layoutparser_ort::LayoutModel::from_pretrained()`, processes each page for layout detection returning `Vec<BBox>`, then applies reading order logic. This architecture achieves end-to-end document processing without Python dependencies, critical for embedded devices, mobile applications, and systems requiring deterministic behavior impossible with garbage-collected runtimes.

**Pre-trained models** populate Hugging Face hub: `microsoft/layoutlm-base-uncased` (110M params, MIT license, full ONNX support), `microsoft/layoutlmv2-base-uncased` (200M params, CC BY-NC-SA 4.0 non-commercial license, partial ONNX), `microsoft/layoutlmv3-base` (133M params, CC BY-NC-SA 4.0, partial ONNX), `microsoft/layoutxlm-base` (multilingual across 53 languages), `naver-clova-ix/donut-base` (200M+ params, MIT license, OCR-free document understanding), and `juliozhao/DocLayout-YOLO-DocStructBench` (YOLO-based layout detection with full ONNX support). License restrictions critically impact deployment—MIT permits commercial use while CC BY-NC-SA 4.0 restricts to non-commercial applications, forcing production systems to either license commercial alternatives or train custom models.

**Surya** (https://github.com/datalab-to/surya) exemplifies production-ready open-source implementation: complete document AI toolkit with OCR, layout analysis, reading order detection, and table recognition in 90+ languages. Models auto-download from Hugging Face on first use, supporting both GPU and CPU inference. Performance benchmarks on A10 GPU show 0.108808s per page for detection (36 batch size, 16GB VRAM), 0.27319s for layout (32 batch size, 7GB VRAM, 88% mean accuracy), 0.4s for reading order (88% mean accuracy), and 0.0219s for table recognition (64 batch size, 10GB VRAM). Modified EfficientViT architecture handles text detection, modified Donut models with grouped query attention and mixture-of-experts layers handle recognition with UTF-16 decoding, and seq2seq models trained on custom datasets handle reading order. The GPL code license and Modified AI Pubs Open Rail-M model license allow free use for research, personal projects, and startups under $2M revenue—commercial deployments exceeding this threshold require alternative licensing.

**DocLayout-YOLO** (https://github.com/opendatalab/DocLayout-YOLO) builds on YOLOv10 with document-specific optimizations including Global-to-Local Adaptive Perception for multi-scale detection. Pre-trained on DocSynth300K (300K synthetic documents), the model achieves 81.7% AP50 (65.6% mAP) on D4LA and 93.0% AP50 (77.4% mAP) on DocLayNet with pretraining. SDK usage involves loading models via `YOLOv10("path/to/model.pt")` and calling `model.predict()` with configurable image size (typically 1024), confidence threshold (typically 0.2), and device specification. Hugging Face hosts pretrained models at `juliozhao/DocLayout-YOLO-DocStructBench` with direct loading support, and real-time inference capabilities make this suitable for interactive applications.

## Computational efficiency determines deployment feasibility

Inference speed benchmarks reveal dramatic performance differences. **Surya on A10 GPU** processes complete document AI pipelines at 0.108s for detection, 0.13s for layout analysis, 0.4s for reading order, and 0.62s for OCR (512 batch size, 20GB VRAM)—total end-to-end processing under 2 seconds per page including all modalities. **GLAM's graph-based approach** achieves 10ms per page (98 pages/second) for layout analysis, 68.7× faster than LayoutLMv3's 687ms and 5.6× faster than YOLOv5x6's 56ms, though with 68.6% mAP versus 76.8% for YOLOv5x6. When speed matters more than the last few percentage points of accuracy, graph methods prove compelling. **MinerU with GPU** achieves 0.21 seconds per page (fastest among document extraction tools), 0.49s for Docling with GPU, 0.86s for Marker with GPU, and 3.3s for MinerU CPU-only—demonstrating that GPU acceleration provides 10-15× speedup for transformer-based systems.

Memory requirements constrain deployment options. Typical VRAM for FP32 inference: LayoutLM-base requires 4-6GB, LayoutLMv2 needs 6-8GB, Donut-base demands 8-10GB, YOLO layout models use 2-4GB, and Surya's full stack peaks at 20GB for maximum batch sizes. INT8 quantization halves these requirements: LayoutLM-base drops to 2-3GB quantized, LayoutLMv2 to 3-4GB, Donut-base to 4-5GB, YOLO to 1-2GB. CPU RAM requirements roughly double GPU VRAM needs for similar performance—LayoutLM-base needs ~8GB RAM, LayoutLMv2 ~12GB, Donut-base ~16GB, and Surya full stack ~32GB.

Hardware recommendations scale with requirements. **Minimum CPU-only configurations** use 8+ cores (Intel i7/AMD Ryzen 7 or better), 16GB RAM, and SSD storage achieving 5-15 pages per minute—acceptable for batch processing but inadequate for interactive applications. **Recommended GPU configurations** deploy NVIDIA RTX 3060 (12GB) or better with 4+ cores CPU, 16GB RAM, and 8-12GB VRAM minimum, achieving 30-100 pages per minute suitable for production services with moderate throughput. **Production high-throughput systems** leverage NVIDIA A10 (24GB), A100 (40/80GB), or H100 GPUs with 16+ cores CPU and 64+ GB RAM, using multiple GPUs for parallel processing to achieve 200+ pages per minute per GPU—necessary for services processing millions of documents.

Model compilation provides substantial speedups. Surya enables compilation via environment variables `COMPILE_DETECTOR=true`, `COMPILE_LAYOUT=true`, `COMPILE_TABLE_REC=true`, or `COMPILE_ALL=true`, achieving 3.3% faster detection, 0.9% faster layout, and 11.5% faster table recognition on A10 GPU. While modest percentages, these gains compound for high-throughput systems processing millions of pages. PyTorch's TorchScript compilation, ONNX graph optimization (operator fusion, constant folding), and ONNX Runtime execution providers (CUDA, TensorRT, DirectML) with session options tuning and I/O binding reduce overhead further—production deployments commonly achieve 2× additional speedup through careful runtime optimization.

The accuracy-speed tradeoff manifests clearly: seq2seq LayoutReader achieves very high accuracy but slow inference at ~0.13s per image due to autoregressive decoding with O(n²) complexity, consuming high memory storing full sequence representations—best for offline processing requiring maximum accuracy. Parallel classification via LayoutLMv3 token classification trades slightly lower accuracy for fast single-pass inference with moderate memory—optimal for production systems and real-time applications. Heuristic XY-Cut methods achieve 61.7% PER (poor on complex layouts) but very fast geometric sorting with minimal memory—suitable only for simple documents or prototyping. Graph neural networks balance competitive accuracy with moderate speed (graph construction plus inference) and moderate memory—effective for relationship-heavy documents where interpretability matters.

## Future directions and practical recommendations

The field advances toward unified architectures handling detection, reading order, and hierarchical structure simultaneously within single models, eliminating task-specific heads and enabling generalist document understanding. Efficiency improvements through FastViT-style hybrid architectures, selective computation based on document complexity, and dynamic token allocation promise to close the speed gap between accurate transformers and fast heuristics. Zero-shot capabilities via vision-language models with document instruction tuning enable transfer across document types without fine-tuning and few-shot adaptation to new layouts, reducing the annotation burden for specialized domains.

**For practitioners implementing reading order detection in 2025**, the recommended path depends on requirements. Systems prioritizing maximum accuracy should deploy LayoutReader or MinerU's VLM backend, accepting the computational cost. Production systems balancing accuracy and speed benefit from YOLOv5x6 layout detection combined with parallel LayoutLMv3-based reading order classification, achieving 70-80% end-to-end accuracy at 30-100 pages per minute. Multi-column document specialists should adopt XY-Cut++ achieving 98.8% BLEU through hybrid heuristic-ML approaches. Edge and embedded deployments requiring minimal resources use graph-based methods like GLAM processing 98 pages per second with 4M parameters, or Rust implementations with layoutparser-ort eliminating Python runtime overhead entirely.

Dataset selection critically impacts training outcomes: scientific documents benefit from PubLayNet's 360K homogeneous layouts achieving 93%+ mAP, diverse layouts require DocLayNet's 80K manually annotated pages despite 10% lower ceiling due to inherent ambiguity, and reading order focus demands ReadingBank's 500K automated annotations from structured WORD documents. Training strategies should leverage pre-trained LayoutLM when possible (MIT license for v1 enabling commercial use), use document-wise splitting preventing overfitting to templates, and recognize that 80% of full data often suffices—focus on diversity over size. Fine-tuning LayoutLMv3-base on domain-specific data typically outperforms training larger models from scratch, and ensembling graph-based spatial reasoning with transformer visual understanding achieves state-of-the-art results exceeding either approach alone.

Open research questions remain: handling reading order in extremely complex layouts like magazines and newspapers with 5+ columns and non-rectangular text flow, optimal token budget allocation between visual and textual modalities in multimodal transformers, universal evaluation metrics for reading order across different document types (BLEU penalizes creative layouts that may have multiple valid orders), integration of human feedback for ambiguous cases through active learning, and handling dynamic/interactive document formats like responsive web pages and adaptive PDFs. The 10-15% gap between current systems (72-77% mAP) and human inter-annotator agreement (82-83% mAP) on diverse layouts represents both the realistic performance ceiling due to genuine ambiguity and the opportunity space for architectural improvements.

**Production deployment** follows established patterns: containerize with Docker pre-downloading models during build (avoid runtime downloads), use persistent volumes for model cache, configure CPU threads via `OMP_NUM_THREADS` (typically half available cores), and use CUDA base images for GPU support. Expose models via FastAPI REST endpoints for stateless horizontal scaling behind load balancers, deploy on Kubernetes with GPU resource limits enabling auto-scaling based on queue depth, and monitor inference latency, throughput, GPU/CPU utilization, memory consumption, queue depth, and error rates via Prometheus and Grafana. Scale horizontally with multiple replicas for stateless designs, vertically with multiple GPUs per instance and batch processing for throughput, optimize costs through auto-scaling, spot instances for batch processing, CPU fallback for low-priority tasks, and INT8/INT4 quantization reducing VRAM requirements by 2-4×.

The convergence on ONNX as the production inference format, Rust for systems requiring performance and minimal dependencies, and quantization for deployment at scale establishes a clear path from research to production. Open-source implementations like Surya, DocLayout-YOLO, and layoutparser-ort demonstrate that production-ready reading order detection no longer requires proprietary systems—though commercial solutions like Mathpix still maintain a narrow accuracy lead through larger training budgets and proprietary data. The democratization of document AI through open datasets (ReadingBank, DocLayNet), open models (LayoutLM family, YOLO variants), and open implementations enables practitioners to deploy state-of-the-art reading order detection matching or exceeding commercial alternatives from 2-3 years ago, with the research-to-production lag decreasing as ONNX export and quantization become standard practices.
